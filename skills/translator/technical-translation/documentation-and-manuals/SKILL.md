# Documentation And Manuals

Technical documentation and user manuals are instructions that people follow to operate, maintain, install, configure, or repair products and systems. The translation must be procedurally accurate, because users act on it. A mistranslated step, a skipped warning, or an inconsistent term can cause the user to perform the wrong action, damage equipment, void a warranty, or injure themselves. Technical translation also demands consistency, because the same component, command, or status must be named the same way across the manual, the interface, and any related materials, or users cannot map instructions to what they see. The work is less about elegant prose than about clarity, completeness, and exact correspondence between the source's procedures and the target's procedures. Errors are often invisible to the translator, who is not operating the device, and visible only to the user who follows the wrong instruction.

Use this skill when translating user manuals, technical documentation, online help, installation guides, maintenance procedures, or reference documentation, or when ensuring translated documentation is clear, complete, and safe. The goal is to produce target documentation that users can follow correctly and safely, with every procedure, warning, and specification preserved.

## Core Rules

### Preserve Procedural Accuracy Above All

The primary function of documentation is to enable correct action. Preserve procedural accuracy as the highest priority.

Every step, sequence, condition, and result must correspond exactly to the source. Do not reorder steps for flow, because the order often reflects a required sequence. Do not combine or split steps without confirming the procedure remains correct. Preserve conditional logic, such as if this happens, do that. Preserve the relationship between actions and their expected results, because users use results to confirm they performed the action correctly. Preserve notes, cautions, warnings, and dangers, and their placement relative to the steps they protect.

When a procedure seems unclear or unsafe in the source, flag it rather than smoothing it, because the issue may be a source defect that affects users in all languages.

### Maintain Terminology Consistency Across All Materials

Technical documentation lives in an ecosystem with the product's interface, marketing, training, and support content. Terminology must be consistent across all of it.

Build and apply a termbase covering component names, commands, status indicators, error messages, and interface labels. Use the same term for the same concept everywhere. Coordinate with the localization of the interface so that manual terms match UI terms exactly, because users move between them. When the interface uses one term and the manual uses another, users cannot follow instructions.

Inconsistency is the most common and most damaging documentation localization defect, because it directly impedes usability.

### Handle Warnings Cautions And Safety Information

Technical documentation contains safety information whose accuracy and prominence protect users. Handle it with care.

Preserve the distinction between danger, warning, caution, and notice, because these signal words carry defined levels of hazard. Map them to the target language's equivalent signal words according to the relevant standard. Preserve the placement of safety information relative to the steps it protects, because a warning after the dangerous step is too late. Preserve formatting such as bold, symbols, and color that indicate severity. Do not soften safety language for tone.

A safety warning that loses its level, placement, or prominence fails to protect the user.

### Preserve Specifications Numbers And Units

Specifications, measurements, tolerances, and units are factual content that must be carried exactly. Errors cause incorrect operation and equipment damage.

Preserve exact values and units. Convert units only where the source or target market requires conversion, and document conversions. Watch for locale-specific decimal and thousands separators. Verify part numbers, model numbers, torque values, voltages, pressures, and other specifications against the source. Ensure specifications in the documentation match the product's actual labeling and interface.

A wrong torque value or voltage in translated documentation can damage equipment or injure users.

### Handle Figures Callouts And Cross-References

Documentation is full of figures, tables, callouts, and cross-references. Translate and align them carefully.

Translate figure captions, table headings, callout labels, and legend text. Ensure callout numbers in figures match the text that references them. Preserve cross-references to sections, figures, tables, and steps so they resolve correctly in the target. Where figures contain embedded text, coordinate localization of the figure, because a figure left in the source language inside translated documentation looks unfinished and can confuse users.

Misaligned callouts and broken cross-references make documentation unusable.

### Adapt Format And Structure For The Target Language

Documentation formatting must accommodate the target language's characteristics. Plan for expansion, contraction, and script differences.

Text expansion, often 20 to 30 percent for some language pairs, can break layouts, tables, and line lengths. Plan for it, especially in tables, callouts, and tight layouts. For non-Latin scripts, account for character width, line height, and font support. For right-to-left languages, mirror layouts and ensure direction is set correctly. Coordinate with desktop publishing or engineering to adjust layouts that the translation breaks.

Documentation that reads correctly but whose layout is broken is hard to use and looks unprofessional.

### Verify Completeness Against The Source

Documentation localization often drops content silently. Verify completeness systematically.

Check that all sections, steps, notes, warnings, figures, tables, captions, headers, footers, and appendices are translated. Hidden content such as metadata, alt text, and table of contents entries must be included. Run completeness checks comparing source and target segment counts and flag discrepancies. A manual missing a section or a step is incomplete and potentially unsafe.

### Support User Testing For Critical Procedures

For documentation covering safety-critical or complex procedures, user testing in the target language reveals whether users can follow it. Recommend or support this testing.

Linguistic review confirms the text is correct. User testing confirms users can act on it. For procedures where failure causes harm or damage, testing catches instructions that are technically accurate but practically confusing. Flag where testing is warranted and refine content based on results.

## Common Traps

### Reordering Or Combining Steps For Flow

Changing the sequence or granularity of steps can make the procedure incorrect and unsafe.

### Inconsistent Terminology Across Manual And Interface

Using different terms for the same component in the manual and the UI prevents users from following instructions.

### Softening Or Misplacing Safety Warnings

Warnings that lose severity, placement, or prominence fail to protect users from hazards.

### Errors In Specifications And Units

Wrong values, units, or separators cause incorrect operation and equipment damage.

### Misaligned Callouts And Broken Cross-References

Figures and references that do not match the text make documentation unusable.

### Ignoring Layout Expansion

Text expansion that breaks tables and layouts produces documentation that looks unprofessional and is hard to use.

### Silently Dropping Content

Missing sections, steps, or hidden content makes the documentation incomplete and potentially unsafe.

### Skipping User Testing For Critical Procedures

Accurate text that users cannot follow causes errors; testing catches practical confusion.

## Self-Check

Before approving translated documentation, verify:

- Procedural accuracy is preserved, with every step, sequence, condition, and result matching the source.
- Terminology is consistent across the manual, the interface, and related materials, governed by a termbase.
- Warnings, cautions, dangers, and notices preserve their level, placement, prominence, and formatting.
- Specifications, numbers, units, part numbers, and tolerances are exact, with conversions documented and locale separators checked.
- Figures, callouts, table headings, captions, and cross-references are translated and aligned so references resolve correctly.
- Format and structure accommodate the target language's expansion, script, and direction, with broken layouts fixed.
- Completeness is verified against the source, including hidden content such as metadata, alt text, and table of contents.
- User testing is recommended or supported for safety-critical or complex procedures.
- No step, warning, specification, or reference has been altered, omitted, or misplaced in a way that affects correct or safe use.
- The translated documentation, followed by a target user, would lead to correct and safe operation of the product.
