# Style Rule Application

A style guide is only valuable when its rules are actually applied, and application is harder than it looks. Style rules are written as general principles, but translation happens one segment at a time, and the gap between an abstract rule and a concrete decision in a real sentence is where most compliance fails. Rules conflict with each other, conflict with naturalness, conflict with terminology requirements, and conflict with the source's structure, and the translator must resolve those conflicts in context without a rulebook that anticipates every case. Style rule application is the work of operationalizing abstract rules into segment-level decisions, resolving the conflicts that arise, and verifying that the result complies rather than merely feeling compliant. The harm this skill prevents is content that was translated with the style guide open but still drifts from it, because the rules were applied selectively, interpreted loosely, or sacrificed silently whenever they collided with convenience.

Agents miss this work because applying a style guide feels like following instructions. It is not. Rules underdetermine decisions, they conflict, and they carry implicit priority that must be inferred, so faithful application is a series of judgments, not a lookup.

## Core Rules

### Read The Governing Style Guide Before Translating, Not During

Internalize the governing style guide before drafting, so its rules shape decisions from the first segment rather than being retrofitted during review. Identify which guide governs and which version, because a project may carry a client guide, a domain guide, and a house guide that disagree. Confirm precedence before working, so when rules conflict you already know which wins.

Translating first and checking style later produces content that conforms only where conformance was easy and drifts wherever the natural rendering happened to violate a rule the translator did not recall. Front-load the rules.

### Operationalize Each Rule Into A Concrete Decision Pattern

Abstract rules underdetermine segment-level choices, so convert each rule into a concrete decision pattern you can apply. A rule favoring the active voice becomes a pattern: identify the agent, make it the subject, use the active verb, unless the agent is unknown or emphasis belongs on the action. A rule about sentence length becomes a threshold and a revision move: if a sentence exceeds the threshold, split at a logical boundary. A rule about forms of address becomes a lookup: confirm the approved pronoun and verb form and apply it consistently.

Operationalizing rules turns them from principles into repeatable actions. A rule that stays abstract gets applied inconsistently, because each segment invites a fresh interpretation.

### Resolve Conflicts Between Rules By Precedence And Intent

Style rules conflict in real text. A rule favoring brevity may conflict with a rule favoring clarity; a rule favoring directness may conflict with a rule favoring politeness; a terminology requirement may conflict with a style rule about word choice. Resolve conflicts by precedence first: the established priority among guides, and within a guide, the explicit hierarchy if one exists. Where precedence does not decide, resolve by intent: determine which rule's purpose is more important in this context, and document the choice.

Never resolve a conflict by ignoring one rule silently. Silent resolution is how style guides erode, because the exceptions become the de facto practice and nobody records why.

### Handle The Conflict Between Style And Naturalness

A recurring conflict is between a style rule and natural target-language expression. A rule imported from the source language may produce awkward target prose; a rule about structure may fight the target language's syntax. Decide deliberately: follow the rule where it serves clarity, consistency, or brand identity, and allow controlled deviation where the rule would produce unnatural or incorrect target text. Document the deviation and its basis.

The decision criterion is whether following the rule harms readability or correctness enough to outweigh the consistency benefit. Where the harm is real, deviate with rationale; where it is merely a preference for a different rendering, follow the rule.

### Handle The Conflict Between Style And Terminology

Terminology often overrides style, because a controlled term must be used even when it violates a style preference. A style rule favoring short words does not override a required multiword technical term. A rule about avoiding repetition does not override the need to use the same approved term consistently. Establish that terminology takes precedence over stylistic preference, and apply style rules around the terminology rather than against it.

Record the precedence so reviewers do not flag the terminology as a style violation, which is a common and wasteful review loop.

### Apply Rules Consistently, Not Selectively

The most common application failure is selective compliance: rules are followed where convenient and ignored where they require effort. Apply each governing rule across the whole content, not only where it is easy. Selective application produces content that looks styled in places and unstyled in others, which reads as carelessness.

Track which rules you have applied and verify coverage at the end, rather than assuming compliance because the rules were in mind during drafting.

### Verify Compliance Against The Guide, Not Against Impression

After drafting, verify compliance explicitly against the style guide, surface by surface, rather than relying on the impression that the text reads well. A fluent translation can violate multiple style rules without feeling wrong. Check each governed surface: forms of address, voice, sentence conventions, punctuation, capitalization, formatting, and terminology treatment. Use a checklist derived from the guide.

Impression-based verification passes content that violates rules the translator did not think to check. Checklist-based verification forces coverage of every rule the guide imposes.

### Flag Rules That Are Ambiguous Or Unworkable

When a rule is too ambiguous to apply consistently or unworkable in the target context, flag it rather than guessing. Feed the ambiguity back to the style guide owner as a clarification request, so the rule is improved for future work. Applying an ambiguous rule by private interpretation means every translator interprets it differently, and the guide's authority erodes.

Flagging also protects the translator: a documented clarification request shows the issue was surfaced, not ignored, and prevents the inconsistency from being blamed on the translator.

## Common Traps

### Checking Style After Translating

Retrofitting style during review conforms only where easy and drifts elsewhere. Read and internalize the governing guide before drafting.

### Leaving Rules Abstract

Abstract principles get interpreted fresh each segment, producing inconsistency. Operationalize each rule into a concrete decision pattern.

### Silent Conflict Resolution

Ignoring one rule to resolve a conflict erodes the guide and hides the de facto practice. Resolve by precedence and intent, and document.

### Following Rules Into Unnatural Text

Imposing a rule that harms target readability or correctness produces styled-but-awkward content. Allow controlled deviation with rationale where harm is real.

### Letting Style Override Terminology

Controlled terms must be used even when they violate style preferences. Establish terminology precedence and apply style around it.

### Selective Compliance

Following rules only where convenient reads as carelessness. Apply each rule across the whole content and verify coverage.

### Impression-Based Verification

Fluent text can violate rules without feeling wrong. Verify compliance against a checklist derived from the guide, surface by surface.

### Private Interpretation Of Ambiguous Rules

Guessing at ambiguous rules means every translator decides differently. Flag ambiguities and request clarification from the guide owner.

## Self-Check

Before approving translated content as style-compliant, verify:

- The governing style guide and its version were read and internalized before drafting, with precedence confirmed among multiple guides.
- Each style rule was operationalized into a concrete, repeatable decision pattern rather than left as an abstract principle.
- Conflicts between rules were resolved by precedence and intent, with the resolution documented rather than one rule silently ignored.
- Conflicts between style and naturalness were decided by whether the rule's harm to readability or correctness outweighed its consistency benefit, with deviations documented.
- Terminology was confirmed to take precedence over stylistic preference, and style rules were applied around controlled terms.
- Each governing rule was applied across the whole content, not only where convenient, and coverage was tracked.
- Compliance was verified against a checklist derived from the guide, surface by surface, not by impression of fluency.
- Ambiguous or unworkable rules were flagged to the guide owner for clarification rather than interpreted privately.
- No rule was followed into unnatural or incorrect target text, and no controlled term was sacrificed to a style preference.
- The content would pass the same compliance check if repeated by another translator using the same guide and precedence.
