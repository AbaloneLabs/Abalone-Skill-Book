# Project Communication Plan Design

A project without a communication plan generates communication anyway, but it is ad hoc, redundant, and inconsistent. Some stakeholders get too much; others get too little; the same information is repeated across five channels; critical updates fall through the cracks. A deliberate communication plan brings order by specifying what information each audience needs, in what format, at what frequency, and through what channel. The judgment problem is that communication planning is often skipped or treated as a formality, leaving the project to default to whatever meetings and reports emerge organically. The result is noise without clarity, and effort without alignment.

Use this skill before creating a communication plan, before establishing project documentation standards, when communication has become chaotic or redundant, or when stakeholders complain of being uninformed despite high communication volume. The goal is to prevent the agent from defaulting to ad hoc communication, over-communicating low-value information, under-communicating critical updates, or fragmenting information across too many channels.

## Core Rules

### [ ] Base The Plan On Stakeholder Information Needs

A communication plan is not a list of reports the project wants to produce; it is a mapping of what each stakeholder group needs to know to perform its role. Start from the stakeholder analysis and build communication around those needs. Information that no one needs is noise.

- [ ] Derive communication requirements from the stakeholder analysis.
- [ ] Specify what each audience needs to know, why, and to what end.
- [ ] Eliminate communication that serves no stakeholder need.
- [ ] Prioritize information that drives decisions or action.

### [ ] Define A Communication Matrix

A communication matrix codifies who receives what, when, and how. It lists each communication type, its audience, format, frequency, owner, and channel. The matrix makes the plan explicit, prevents gaps and overlaps, and gives the team a shared reference.

- [ ] Create a matrix with columns for type, audience, format, frequency, owner, channel.
- [ ] Include all communication types: status reports, meetings, dashboards, reviews.
- [ ] Ensure every stakeholder group is covered by at least one communication type.
- [ ] Review and update the matrix as needs change.

### [ ] Match Channel To Message And Audience

Each channel has strengths and weaknesses. Dashboards are good for metrics; meetings are good for discussion and decisions; emails are good for asynchronous updates; instant messaging is good for quick coordination. Mismatched channels, a complex decision buried in an email, a simple status update consuming a meeting, waste time and reduce clarity.

- [ ] Choose channels based on the message's complexity, urgency, and interactivity.
- [ ] Use meetings for discussion, decisions, and alignment, not for one-way updates.
- [ ] Use dashboards and reports for metrics and status.
- [ ] Avoid using instant messaging for decisions that need a record.

### [ ] Right-Size Frequency To Avoid Overload And Gaps

Too frequent, and communication becomes noise that stakeholders ignore; too infrequent, and stakeholders lose visibility and feel uninformed. Frequency should match the stakeholder's need and the project's pace. High-influence, high-interest stakeholders may need weekly updates; others may need monthly.

- [ ] Set frequency based on stakeholder need and project pace.
- [ ] Increase frequency during critical or high-change periods.
- [ ] Decrease frequency during stable periods to reduce noise.
- [ ] Check periodically whether the frequency is right.

### [ ] Establish Documentation Standards

Consistent documentation, standardized report formats, naming conventions, and version control, makes information findable and comprehensible. Without standards, each report looks different, files are scattered, and finding the current version becomes a scavenger hunt.

- [ ] Define standard formats for status reports, meeting notes, and decision logs.
- [ ] Establish naming conventions and a central repository for project documents.
- [ ] Use version control so the current version is always identifiable.
- [ ] Maintain a single source of truth for each document type.

### [ ] Distinguish Push, Pull, And Interactive Communication

Push communication (email, reports) sends information to recipients; pull communication (portals, dashboards) makes information available for recipients to retrieve; interactive communication (meetings, calls) enables dialogue. Each has a role, and the plan should use the right mode for each message.

- [ ] Use push for information stakeholders must receive (alerts, decisions).
- [ ] Use pull for reference information stakeholders access on demand (dashboards, archives).
- [ ] Use interactive for discussion, alignment, and decision-making.
- [ ] Avoid using push for everything, which creates overload and ignoring.

### [ ] Define Escalation Paths Within The Plan

The communication plan should specify how issues and decisions escalate, to whom, and through what channel. Clear escalation paths prevent issues from stalling and ensure the right authority is engaged at the right time.

- [ ] Define escalation levels and the authority at each level.
- [ ] Specify the channel and timeframe for each escalation level.
- [ ] Document who is notified when an issue escalates.
- [ ] Ensure escalation paths are known before they are needed.

### [ ] Plan For Critical And Crisis Communication

Routine communication plans handle normal operations; crises require a different approach. Define in advance how the project communicates during a crisis: who is informed, how quickly, through what channel, and with what message discipline.

- [ ] Define a crisis communication protocol with roles and channels.
- [ ] Specify notification timeframes for critical issues.
- [ ] Establish message discipline to avoid conflicting communications.
- [ ] Test the crisis protocol before it is needed.

### [ ] Avoid Channel Proliferation

Each new channel, a new chat group, a new report, a new tool, fragments attention and increases the chance that information is missed. Consolidate channels where possible and resist adding new ones without retiring old ones.

- [ ] Limit the number of active communication channels.
- [ ] Consolidate redundant channels.
- [ ] Retire a channel when adding a new one.
- [ ] Ensure each channel has a clear purpose known to users.

### [ ] Review And Adjust The Plan Regularly

A communication plan that was right at kickoff may be wrong at month four. Stakeholder needs change, project pace shifts, and some channels prove ineffective. Review the plan periodically and adjust based on feedback and observed effectiveness.

- [ ] Review the communication plan at phase transitions or major changes.
- [ ] Solicit feedback from stakeholders on communication effectiveness.
- [ ] Adjust frequency, channels, and content based on feedback.
- [ ] Retire communication that is no longer useful.

## Common Traps

### [ ] No Plan, Ad Hoc Chaos

Defaulting to organic communication that is inconsistent and redundant.

### [ ] Communication Based On Producer Needs

Producing reports the project wants to send rather than what stakeholders need.

### [ ] Channel Mismatch

Using the wrong channel for the message, wasting time and reducing clarity.

### [ ] Uniform Frequency

Sending all communication at the same frequency regardless of stakeholder need.

### [ ] No Documentation Standards

Inconsistent formats and scattered files that make information hard to find.

### [ ] Push Everything

Overloading stakeholders with pushed information until they stop reading.

### [ ] No Escalation Paths

Issues stalling because escalation authority and channels are undefined.

### [ ] Channel Proliferation

Adding channels without consolidating, fragmenting attention and information.

## Self-Check

- [ ] Is the communication plan based on stakeholder information needs, not producer preferences?
- [ ] Is there a communication matrix specifying audience, format, frequency, owner, and channel?
- [ ] Are channels matched to message complexity, urgency, and interactivity?
- [ ] Is frequency right-sized to avoid both overload and gaps?
- [ ] Are documentation standards established for formats, naming, and version control?
- [ ] Are push, pull, and interactive communication modes used appropriately?
- [ ] Are escalation paths defined with levels, channels, and timeframes?
- [ ] Is there a crisis communication protocol with roles and message discipline?
- [ ] Are channels consolidated to avoid proliferation and fragmentation?
- [ ] Is the plan reviewed and adjusted based on stakeholder feedback and effectiveness?
