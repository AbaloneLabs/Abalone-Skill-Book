# Milestone And Deliverable Tracking

Progress is not measured by hours logged or tasks closed. It is measured by meaningful state changes: a decision made, a deliverable accepted, a gate passed. Milestones make these state changes visible and give the project manager a way to track reality rather than the illusion of percent complete. Without well-defined milestones, a project can report steady progress while drifting toward a missed commitment that no one saw coming.

Use this skill before defining milestones, setting up deliverable tracking, preparing a milestone or gate review, or diagnosing why progress reports do not match reality. The goal is to prevent the agent from tracking activity instead of outcomes and from letting milestones become arbitrary calendar markers.

## Core Rules

### Define Milestones As Meaningful State Changes

A milestone should represent a significant transition in the project, not a date on a calendar. Good milestones mark events such as scope approval, design lock, feasibility proven, contract signed, integration complete, user acceptance passed, launch readiness approved, or handoff complete.

Weak milestones mark time, such as end of week six or month two close, without describing what state the project must reach. A date without a state change is a deadline, not a milestone.

Each milestone should answer what is true when it is achieved that was not true before.

### Attach Entry And Exit Criteria

A milestone is only useful if its completion is verifiable. Define entry criteria, what must be true before work toward the milestone can be considered complete, and exit criteria, what must be true to declare the milestone achieved.

For example, a user acceptance milestone might require entry criteria of feature complete and test environment ready, and exit criteria of acceptance tests passed and signoff received from the business owner.

Without criteria, milestone completion becomes a negotiation or a political decision rather than an objective check.

### Assign A Single Accountable Owner

Each milestone needs one accountable owner who is responsible for ensuring the exit criteria are met. This is not necessarily the person doing all the work, but it is the person who owns the outcome and who will be asked if the milestone slips.

Diffuse ownership produces diffuse accountability. When several people are jointly responsible, no one is.

### Track Deliverables, Not Just Dates

Deliverables are the tangible outputs that prove progress. Track the status of each deliverable against its acceptance criteria, its owner, its dependencies, and its target milestone.

A deliverable tracker should show, for each item, whether it is not started, in progress, under review, accepted, or blocked. This is more informative than a percent complete figure that can hide blocked or incomplete work.

### Use Milestones As Decision Gates

Milestones are natural points to decide whether to proceed, adjust, or stop. A gate review at a milestone should evaluate whether exit criteria are met, whether risks have changed, whether the business case still holds, and whether the next phase is still justified.

Gates prevent the sunk-cost fallacy of continuing simply because work has begun. A gate that always approves is not a gate.

### Forecast Milestone Achievement Honestly

For each upcoming milestone, forecast whether it will be achieved on time and with what confidence. Base the forecast on completed prerequisites, remaining critical path work, open decisions, and external dependencies.

Distinguish a committed milestone from a forecast or at-risk milestone. If a milestone is at risk, say so early enough for action, not when it has already slipped.

### Review And Update Milestone Status Regularly

Milestone status is not set once. Review milestones at a regular cadence, update their forecast, record movement, and escalate milestones that are slipping. A milestone register that is never updated is just decoration.

When a milestone moves, record why it moved, what tradeoff is proposed, and what the new forecast is.

## Common Traps

### Calendar Markers Mistaken For Milestones

A date without a state change does not track progress; it marks time.

### No Verifiable Criteria

Without entry and exit criteria, milestone completion is opinion and negotiation.

### Diffuse Ownership

Joint accountability means no accountability when the milestone slips.

### Percent Complete Instead Of Deliverable Status

Percent figures hide blocked work and create false comfort. Track accepted deliverables.

### Gate Reviews That Always Approve

A gate that never stops or adjusts the project provides no control and wastes the review.

### Late Disclosure Of Slippage

Holding bad news until the milestone date prevents stakeholders from helping with tradeoffs.

### Milestones Without Dependencies Mapped

A milestone that depends on an unstarted prerequisite is already at risk, and not mapping the dependency hides that.

### Stale Milestone Register

A register created at kickoff and never updated stops reflecting reality and stops being trusted.

## Self-Check

- [ ] Each milestone represents a meaningful state change, not merely a calendar date.
- [ ] Entry and exit criteria are defined for every milestone and are objectively verifiable.
- [ ] Each milestone has a single accountable owner responsible for the outcome.
- [ ] Deliverables are tracked by status against acceptance criteria, not only by percent complete.
- [ ] Milestones are used as decision gates that can adjust or stop the project, not rubber stamps.
- [ ] Upcoming milestones carry an honest forecast and confidence level based on prerequisites and dependencies.
- [ ] Committed, forecast, and at-risk milestones are distinguished.
- [ ] Milestone status is reviewed at a regular cadence and updated as conditions change.
- [ ] Milestone movement is recorded with the reason and the proposed tradeoff.
- [ ] Dependencies affecting each milestone are mapped so risk is visible before the milestone date.
