# Scope Decomposition And WBS Design

A scope statement describes what the project will deliver; a work breakdown structure (WBS) decomposes that scope into manageable, estimable, and assignable components. The WBS is the bridge between scope and execution: it defines all the work, organizes it hierarchically, and provides the structure for estimating, scheduling, and controlling. The judgment problem is that decomposition is both an art and a discipline. Decompose too coarsely and work is missed or unestimable; decompose too finely and the structure becomes unmanageable. The most common failure is a WBS that is organized by activity or phase rather than by deliverable, which leads to missed work, double-counted effort, and a structure that does not support scope control.

Use this skill before creating or revising a WBS, before estimating or scheduling work, when work has been missed or double-counted, or when the WBS does not support effective scope control. The goal is to prevent the agent from creating a WBS that omits work, double-counts effort, is organized by activity rather than deliverable, or is decomposed to an inappropriate level of detail.

## Core Rules

### [ ] Organize The WBS Around Deliverables, Not Activities

A deliverable-oriented WBS decomposes scope into tangible outputs (product components, documents, services), each of which can be verified. An activity- or phase-oriented WBS decomposes into tasks, which are transient and hard to verify, and which invite double-counting and gaps. Deliverable orientation ensures the WBS captures all outputs the project must produce.

- [ ] Structure the WBS around deliverables (tangible, verifiable outputs), not activities.
- [ ] Make each WBS element a product, service, or result, not an action.
- [ ] Use activities only at the work package level, below the WBS, in the schedule.
- [ ] Verify that decomposition produces elements that can be accepted or rejected.

### [ ] Apply The 100% Rule

The 100% rule is the most important WBS principle: the WBS must capture 100% of the project scope, and the sum of work at any level must equal 100% of the parent, no more and no less. This rule catches both omissions (work not represented) and double-counting (work represented in multiple places).

- [ ] Ensure the WBS captures 100% of the defined scope, including project management work.
- [ ] Verify that child elements sum to 100% of each parent, with no gaps or overlaps.
- [ ] Use the 100% rule as a completeness check at every level of decomposition.
- [ ] Include all deliverables: product, supporting, and project management.

### [ ] Decompose To The Work Package Level

The work package is the lowest level of the WBS, the point at which work can be reliably estimated, scheduled, assigned, and controlled. Decomposition should proceed to this level for all significant work, but not beyond it into micro-tasks that belong in the schedule.

- [ ] Decompose to the work package level where work can be estimated and assigned.
- [ ] Ensure each work package has a single accountable owner.
- [ ] Avoid decomposing beyond the work package into schedule activities.
- [ ] Define work packages small enough to estimate reliably (commonly 8/80 or similar rules).

### [ ] Choose The Right Level Of Detail

Over-decomposition creates an unmanageable structure with hundreds of tiny elements; under-decomposition leaves work that is too large to estimate or control. The right level balances manageability against control. Larger or riskier projects may justify finer decomposition.

- [ ] Decompose to a level that supports reliable estimation and control.
- [ ] Avoid over-decomposition that creates unmanageable detail.
- [ ] Avoid under-decomposition that leaves work too large to estimate or assign.
- [ ] Adjust detail level based on risk, uncertainty, and project size.

### [ ] Include Project Management And Supporting Deliverables

The WBS often omits project management work (planning, monitoring, closing) and supporting deliverables (training, documentation, transition), focusing only on the product. The 100% rule requires all work to be represented, including these categories, or it will be unstaffed and unbudgeted.

- [ ] Include project management deliverables (plans, reports, reviews) in the WBS.
- [ ] Include supporting deliverables (documentation, training, deployment).
- [ ] Include integration, testing, and closure work.
- [ ] Verify that non-product work is not omitted.

### [ ] Use A Consistent Decomposition Approach

Mixed decomposition approaches, some branches by deliverable, others by phase or team, create confusion and violate the 100% rule. Apply a consistent decomposition criterion, typically deliverable, across the entire WBS.

- [ ] Apply a single decomposition criterion (deliverable) consistently across the WBS.
- [ ] Avoid mixing deliverable-oriented and phase-oriented branches.
- [ ] Ensure siblings at each level represent the same type of decomposition.
- [ ] Document the decomposition approach for consistency.

### [ ] Make The WBS The Foundation For Estimating And Scheduling

The WBS is not an end in itself; it is the structure against which estimates are built, the schedule is organized, and costs are collected. Each work package should map to schedule activities and cost accounts, ensuring integration across the plan.

- [ ] Base estimates on work packages, not on higher-level WBS elements.
- [ ] Organize the schedule around work packages and their activities.
- [ ] Map cost accounts to WBS elements for cost control.
- [ ] Ensure traceability from scope to WBS to schedule to budget.

### [ ] Use A WBS Dictionary For Complex Elements

For complex or ambiguous work packages, a WBS dictionary defines the work, its boundaries, acceptance criteria, and responsible party. The dictionary prevents ambiguity about what each element includes and excludes, reducing scope disputes.

- [ ] Create a WBS dictionary for complex or ambiguous work packages.
- [ ] Define each element's scope, boundaries, and acceptance criteria.
- [ ] Specify what is included and what is excluded.
- [ ] Use the dictionary to resolve scope questions during execution.

### [ ] Keep The WBS Aligned With The Scope Baseline

The WBS is a component of the scope baseline. Changes to scope must flow through change control and update the WBS. A WBS that drifts from the approved scope loses its value as a control mechanism.

- [ ] Maintain the WBS as part of the scope baseline.
- [ ] Update the WBS through formal change control when scope changes.
- [ ] Preserve baseline versions to track scope evolution.
- [ ] Ensure the current WBS reflects the current approved scope.

### [ ] Validate The WBS With The Team And Stakeholders

A WBS created by the project manager alone reflects one perspective and misses work visible to others. Validate the WBS with the delivery team and key stakeholders to catch omissions and confirm completeness.

- [ ] Review the WBS with the delivery team for completeness and accuracy.
- [ ] Validate with stakeholders that all expected deliverables are represented.
- [ ] Use team review to catch omissions and ambiguities.
- [ ] Iterate the WBS based on review feedback.

## Common Traps

### [ ] Activity-Oriented WBS

Organizing by tasks or phases rather than deliverables, causing gaps and double-counting.

### [ ] Violating The 100% Rule

Allowing gaps (omitted work) or overlaps (double-counted work) at any level.

### [ ] Over-Decomposition

Creating hundreds of tiny elements that are unmanageable and costly to maintain.

### [ ] Under-Decomposition

Leaving work packages too large to estimate or control reliably.

### [ ] Omitting Non-Product Work

Leaving project management and supporting deliverables out of the WBS.

### [ ] Mixed Decomposition Approaches

Combining deliverable-oriented and phase-oriented branches inconsistently.

### [ ] WBS Disconnected From Schedule And Budget

Failing to map work packages to estimating, scheduling, and cost control.

### [ ] Single-Perspective Creation

Building the WBS without team validation, missing work visible to others.

## Self-Check

- [ ] Is the WBS organized around deliverables (tangible, verifiable outputs), not activities?
- [ ] Does the WBS satisfy the 100% rule at every level, with no gaps or overlaps?
- [ ] Is work decomposed to the work package level where it can be estimated and assigned?
- [ ] Is the level of detail appropriate, neither over- nor under-decomposed?
- [ ] Are project management and supporting deliverables included in the WBS?
- [ ] Is a consistent decomposition approach applied across the entire WBS?
- [ ] Is the WBS the foundation for estimating, scheduling, and cost control?
- [ ] Is a WBS dictionary used for complex or ambiguous elements?
- [ ] Is the WBS maintained as part of the scope baseline and updated through change control?
- [ ] Has the WBS been validated with the delivery team and stakeholders for completeness?
