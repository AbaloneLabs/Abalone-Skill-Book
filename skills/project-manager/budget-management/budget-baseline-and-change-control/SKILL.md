# Budget Baseline And Change Control

A budget becomes controllable only when it is baselined: formally approved as the reference against which spending is measured. Without a baseline, "on budget" is a subjective assertion, and spending drifts silently because there is no fixed point of comparison. The judgment problem is that baselining and change control are often treated as paperwork rather than as the mechanisms that make budget management possible. A baseline that is silently revised whenever spending diverges provides no control; a baseline that is never updated despite approved changes becomes irrelevant. The skill is establishing a meaningful baseline, controlling changes to it formally, and managing reserves so that the budget remains a reliable management tool throughout the project.

Use this skill before establishing a cost baseline, before managing budget changes, when setting up reserve management, or when diagnosing why budget control was ineffective. The goal is to prevent the agent from operating without a baseline, from silently revising it, from confusing contingency with management reserves, or from allowing scope changes without corresponding budget updates.

## Core Rules

### [ ] Establish A Formal Cost Baseline

The cost baseline is the approved, time-phased budget against which performance is measured. It includes all planned costs and reserves allocated to the project. Establish it formally through approval, and use it as the single reference for budget measurement.

- [ ] Create a time-phased cost baseline covering all planned expenditures.
- [ ] Obtain formal approval of the baseline before execution begins.
- [ ] Use the baseline as the reference for all variance measurement.
- [ ] Document the basis of estimate and assumptions behind the baseline.

### [ ] Distinguish Contingency Reserve From Management Reserve

Contingency reserve covers identified risks; management reserve covers unknown unknowns. Contingency is under the project manager's control and allocated to specific risks; management reserve is held at a higher level for unforeseen events. Confusing them undermines reserve discipline.

- [ ] Allocate contingency reserve for identified risks, under PM control.
- [ ] Hold management reserve for unknown unknowns at the organizational level.
- [ ] Do not use contingency for scope changes or management reserve for identified risks.
- [ ] Track reserve usage separately from the base budget.

### [ ] Control Baseline Changes Formally

Changes to the baseline, whether from scope changes, estimate revisions, or approved transfers, must flow through formal change control with documented approval. Silent revisions destroy the baseline's value as a reference. Every change must be evaluated, approved, and recorded.

- [ ] Route all baseline changes through formal change control.
- [ ] Evaluate each change's cost, schedule, and scope impact before approval.
- [ ] Document the approval, rationale, and revised baseline.
- [ ] Never silently revise the baseline to match actual spending.

### [ ] Re-Baseline Only Through Approved Change

Re-baselining replaces the original baseline with a new one. It is appropriate when approved changes fundamentally alter the project, not when spending has diverged and the team wants a clean slate. Frequent re-baselining destroys the ability to track performance trends.

- [ ] Re-baseline only when approved changes fundamentally alter the project.
- [ ] Preserve the original baseline for historical comparison.
- [ ] Document the rationale and approval for every re-baseline.
- [ ] Avoid re-baselining to erase unfavorable variance.

### [ ] Manage Reserve Usage Through Defined Approval

Reserves are not slush funds; their use must be approved and tracked. Contingency release requires evidence that the identified risk has been addressed or materialized; management reserve release requires higher-level approval. Uncontrolled reserve use depletes protection silently.

- [ ] Require documented approval for reserve release.
- [ ] Release contingency when the risk is realized or resolved.
- [ ] Release management reserve only through higher-level approval.
- [ ] Track reserve balance and project remaining reserve adequacy.

### [ ] Keep The Baseline Aligned With Scope

Scope changes must flow through to budget changes. A scope increase without a budget update creates an unfunded mandate; a scope reduction without a budget update wastes authorized funds. The baseline must reflect the current approved scope at all times.

- [ ] Update the baseline when scope changes are approved.
- [ ] Ensure scope increases are funded and scope reductions release funds.
- [ ] Avoid executing approved scope changes without baseline updates.
- [ ] Maintain traceability between scope, schedule, and budget baselines.

### [ ] Measure Variance Against The Baseline, Not A Moving Target

Budget variance is measured against the approved baseline, not against a revised plan or the latest forecast. Measuring against a moving target hides whether the project is truly on budget. The baseline is the fixed reference; forecasts are separate.

- [ ] Measure cost variance against the approved baseline.
- [ ] Distinguish baseline measurement from forecast projection.
- [ ] Do not replace the baseline with the forecast for variance measurement.
- [ ] Report both baseline variance and forecast for full transparency.

### [ ] Communicate Baseline Status And Changes Transparently

Stakeholders must understand the current baseline, how spending compares to it, and what changes have been approved. Transparent communication prevents surprises and maintains trust. Hiding baseline changes or unfavorable variance erodes credibility.

- [ ] Report spending against the baseline transparently.
- [ ] Disclose approved baseline changes and their rationale.
- [ ] Report reserve balances and usage.
- [ ] Avoid hiding unfavorable variance or silent revisions.

### [ ] Preserve Historical Baselines For Trend Analysis

Each approved baseline version should be preserved so that performance can be analyzed over time and across changes. Historical baselines reveal whether estimates are improving, whether changes are accumulating, and whether the project is trending toward overrun.

- [ ] Preserve all baseline versions, not just the current one.
- [ ] Analyze performance trends across baseline versions.
- [ ] Use historical baselines to assess estimation accuracy.
- [ ] Track the cumulative effect of changes over the project lifecycle.

### [ ] Connect Budget Control To Scope And Schedule Control

Budget, scope, and schedule baselines are interdependent; a change to one affects the others. Budget change control must be integrated with scope and schedule change control so that the three baselines remain aligned. Treating them independently creates inconsistencies.

- [ ] Integrate budget change control with scope and schedule change control.
- [ ] Ensure changes update all affected baselines consistently.
- [ ] Assess the cross-baseline impact of every change.
- [ ] Maintain alignment across the three baselines throughout the project.

## Common Traps

### [ ] No Formal Baseline

Operating without an approved baseline, making "on budget" meaningless.

### [ ] Silent Revisions

Revising the baseline to match actuals without formal change control.

### [ ] Reserve Confusion

Using contingency for scope changes or management reserve for identified risks.

### [ ] Re-Baselining To Erase Variance

Replacing the baseline to make performance look better rather than reflecting real change.

### [ ] Uncontrolled Reserve Use

Depleting reserves without approval, leaving the project unprotected.

### [ ] Scope-Budget Misalignment

Executing scope changes without updating the budget baseline.

### [ ] Measuring Against A Moving Target

Using forecasts or revised plans instead of the baseline for variance measurement.

### [ ] Hidden Changes

Not disclosing baseline changes or unfavorable variance to stakeholders.

## Self-Check

- [ ] Is there a formally approved, time-phased cost baseline as the measurement reference?
- [ ] Are contingency and management reserves distinguished and controlled separately?
- [ ] Do all baseline changes flow through formal change control with documented approval?
- [ ] Is re-baselining reserved for fundamental approved changes, not variance erasure?
- [ ] Is reserve usage managed through defined approval and tracked?
- [ ] Is the baseline kept aligned with the current approved scope?
- [ ] Is variance measured against the approved baseline, not a moving target?
- [ ] Are baseline status and changes communicated transparently to stakeholders?
- [ ] Are historical baseline versions preserved for trend analysis?
- [ ] Is budget change control integrated with scope and schedule change control?
