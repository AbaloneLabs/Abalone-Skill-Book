# AI Guardrails And Output Validation

An LLM's output is untrusted data. It is generated by a stochastic process with no guarantee of format, factuality, safety, or intent, and the moment it enters your system — written to a database, shown to a user, passed to a downstream tool, or logged — it must be treated with the same skepticism as user input. The judgment problem is that the output *looks* authoritative: it is fluent, structured, and confident, which makes it easy to forget that it may be malformed, harmful, privacy-leaking, or completely off-task. Guardrails are the validation layer that stands between the model and the rest of the system, and their absence is the difference between an AI feature that fails gracefully and one that corrupts state, leaks data, or causes harm on the rare bad output.

Agents tend to miss these problems because the happy path is convincing — the model returns well-formed JSON that matches the schema, says nothing harmful, and answers the question — so the validation layer feels redundant. The harm appears on the tail. The model occasionally wraps JSON in prose or markdown fences, crashing the parser. A user prompt or a retrieved document contains a social security number that the model faithfully echoes into the output and the logs. A content filter with an unmeasured false-positive rate blocks legitimate queries and the team cannot tell. A refusal ("I can't help with that") is rendered to the user as if it were the answer. A degenerate output — empty, repeating, or off-topic — is persisted as a valid record. The judgment problem is to treat model output as adversarial input to your own system and build the validation, filtering, and monitoring that catches what the model gets wrong.

This skill covers output schema validation, content filtering, PII/secret prevention, anomaly detection, and failure handling. It complements the prompt-engineering skill (which reduces bad outputs at the source) and the llm-api-integration skill (which covers the call mechanics).

## Core Rules

### Treat Model Output As Untrusted Input And Validate Before Use

The foundational rule: whatever the model returns is untrusted data entering your system, exactly like user input, and it must be validated before it is used, stored, displayed, or passed downstream:

- **Validate structure before semantics.** If the downstream expects JSON, parse and schema-validate before anything else; reject (or repair) outputs that do not conform. Do not assume the model complied with "respond in JSON."
- **Validate semantics, not just shape.** A well-formed JSON object with a category outside the allowed enum, a date in the future, a negative price, or a foreign-key id that does not exist is invalid even though it parses. Apply business-rule validation after schema validation.
- **Bound and sanitize free-text output.** If the output is shown to users or stored, apply length limits, strip or escape for the rendering context (HTML, SQL, shell), and treat it as you would any user-generated content for injection risks.
- **Never pass model output directly to a tool or executor.** If the model's output drives a side effect (a function call, a shell command, a database write, an API call), validate and authorize it first. An LLM that can invoke tools with unvalidated arguments is a remote code execution surface.

### Define And Enforce An Output Schema, With A Failure Path

Structured output needs an explicit, enforced schema and a defined behavior when validation fails:

- **Use schema-constrained generation where available.** JSON mode, function/tool calling, and grammar-constrained generation make the model emit conformant output far more reliably than prose instructions. Use them as the first line of defense.
- **Validate against a schema on receipt.** Even with constrained generation, validate the parsed output against a schema (JSON Schema, a typed parser) — constrained generation is not a 100% guarantee.
- **Define the failure path.** When validation fails, what happens? Retry with a corrective prompt ("your output was not valid JSON, respond with only the JSON object"), fall back to a rule-based default, or fail gracefully to the user. Never crash, never persist invalid output, never silently drop it.
- **Bound retries.** A persistently-failing output can loop; cap repair attempts and fall back after the cap. Repair retries cost tokens and latency.

### Filter Harmful And Disallowed Content — And Measure The Filter's Error Rates

Content moderation filters (provider-hosted or self-hosted) catch harmful, abusive, or policy-violating output, but a filter is a classifier with false positives and false negatives that must be measured:

- **Choose what categories to filter by policy.** Hate, violence, sexual, self-harm, PII, competitor names, off-topic — the filter policy is a product and legal decision, not a technical default. Write it down.
- **Measure false-positive and false-negative rates.** A filter that blocks legitimate queries (high false positive) degrades the product; one that misses harmful output (high false negative) creates liability. Measure both on a labeled set, and tune thresholds against the acceptable tradeoff.
- **Decide filter placement: input, output, or both.** Filtering the input (the prompt) catches disallowed requests early; filtering the output catches harmful generations. For safety, filter both; for cost, output filtering is the floor.
- **Handle filter verdicts explicitly.** When the filter flags content, decide: block and explain, sanitize and continue, or escalate to review. A filter whose verdict is ignored is decorative.

### Prevent PII And Secret Leakage In Both Directions

LLM features leak sensitive data in two directions: secrets/PII flow *into* the prompt (and then into provider logs and possibly the output), and the model may *emit* PII it saw in context or recalls from training. Both need defenses:

- **Redact PII and secrets before they reach the model.** Scan user input and retrieved documents for PII (emails, phone numbers, national IDs, credit cards) and secrets (API keys, tokens) and redact or mask them before constructing the prompt. What the model never sees, it cannot leak.
- **Scan output for PII before returning or logging.** Even with redaction, the model may emit PII recalled from training or reconstructed from partial context. Scan output and block or mask PII before it leaves the system.
- **Do not log full prompts and outputs by default.** Prompts and outputs may contain PII; logging them in plaintext creates a retention and breach surface. Log metadata (token counts, latency, flags) by default; log full content only where required and with redaction and access controls.
- **Beware the provider's data handling.** Understand whether the provider trains on or retains your prompts; for sensitive workloads, use endpoints that do not retain, or self-host. The pipeline's privacy boundary includes the provider.

### Detect Anomalous And Degenerate Outputs

Beyond schema and content, the model produces a class of outputs that are technically valid but operationally broken. Detect them:

- **Refusals and canned safety responses.** The model may return "I can't help with that" for a legitimate query (over-refusal). Detect refusal patterns and either retry with a clearer prompt or surface a meaningful message rather than rendering the refusal as the answer.
- **Empty or truncated output.** An empty completion, or one that hit `max_tokens` mid-sentence, is not a valid answer. Detect and handle (retry with higher cap, or fail).
- **Repeating or degenerate output.** The model occasionally loops ("the the the...") or repeats a phrase. Detect repetition and treat as a failure.
- **Off-topic or instruction-drift output.** The model may answer a different question than asked, especially under prompt injection. Where feasible, verify the output addresses the input (a relevance/grounding check).
- **Out-of-distribution confidence.** A model that is always confident regardless of correctness is hard to gate; where the task allows, elicit and sanity-check uncertainty signals.

Track the rates of these anomalies; a spike in refusals or truncations after a prompt or model change is an early warning.

### Monitor Token Usage, Cost, And Latency As Guardrails

Operational metrics are guardrails too — they catch runaway cost and degraded performance before they become incidents:

- **Log per-call token counts (input, output, cached) and compute cost.** Aggregate by feature, user, and tenant; alert on anomalies. A bug that loops calls or inflates prompts shows up as a cost spike.
- **Track latency p95/p99 and timeout/retry rates.** Degraded provider performance surfaces here before users report it.
- **Track validation-failure and filter-trigger rates.** A rising validation-failure rate means the model is producing more malformed output (often after a model or prompt change); a rising filter-trigger rate needs investigation (input drift or a too-aggressive filter).

## Common Traps

### Trusting Model Output Without Validation

Passing the model's output directly to a parser, database, or tool because it "usually" conforms, then crashing or corrupting state on the rare malformed output. Validate structure and semantics before any use.

### Passing Unvalidated Model Output To A Tool Or Executor

Letting the model invoke functions, shell commands, or API calls with unvalidated arguments, creating a code-execution or privilege surface. Validate and authorize every tool argument.

### Schema Validation With No Failure Path

Validating the schema but with no defined behavior on failure, so an invalid output crashes, is persisted, or silently dropped. Define retry-with-correction, fallback, or graceful failure, with bounded retries.

### Content Filter With Unmeasured Error Rates

Deploying a moderation filter without measuring false-positive and false-negative rates, so it either blocks legitimate queries or misses harmful output unnoticed. Measure both and tune thresholds.

### PII Or Secrets Flowing Into The Prompt Unredacted

User input or retrieved documents containing PII or secrets reaching the model unredacted, then leaking into provider logs or the output. Redact before constructing the prompt.

### Logging Full Prompts And Outputs In Plaintext

Logging complete prompts and outputs by default, creating a PII retention and breach surface. Log metadata by default; log full content only with redaction and access controls.

### Rendering A Refusal As If It Were The Answer and accepting Empty, Truncated, Or Repeating Output

The model returns a safety refusal ("I can't help with that") and the system displays it as the answer to a legitimate query, with no detection or retry. Detect refusals and handle meaningfully.

Treating an empty completion, a `max_tokens` truncation, or a degenerate repetition as a valid answer. Detect and retry or fail.

### Ignoring The Provider's Data Retention and no Monitoring Of Validation-Failure Or Filter Rates

Sending sensitive prompts to a provider endpoint that retains or trains on them, expanding the privacy boundary unintentionally. Use non-retaining endpoints or self-host for sensitive workloads.

Shipping without tracking validation-failure, filter-trigger, refusal, and cost rates, so a model or prompt change that degrades output quality goes unnoticed until users complain. Monitor these as guardrails.

## Self-Check

- [ ] All model output is validated as untrusted input: structure (schema/parse) then semantics (enums, ranges, business rules, existence) before any use, storage, display, or tool invocation.
- [ ] No model output reaches a tool, executor, database write, or API call without validation and authorization; tool arguments are treated as a code-execution surface.
- [ ] An output schema is enforced (via constrained generation where available plus schema validation on receipt), with a defined failure path (bounded repair retry, fallback, or graceful failure) that never crashes, persists invalid output, or silently drops it.
- [ ] Content moderation filters cover the policy-defined categories, are placed on input and/or output deliberately, and their false-positive and false-negative rates are measured and tuned against an acceptable tradeoff.
- [ ] PII and secrets are redacted from user input and retrieved documents before reaching the model, output is scanned for PII before return, and full prompts/outputs are not logged in plaintext by default (metadata logging with redaction and access controls instead).
- [ ] The provider's data retention/training policy is understood, and sensitive workloads use non-retaining endpoints or self-hosting.
- [ ] Anomalous and degenerate outputs are detected and handled: refusals, empty/truncated, repeating, and off-topic outputs are not rendered or persisted as valid answers.
- [ ] Operational guardrails are monitored: per-call token counts and cost (by feature/user/tenant), latency p95/p99, timeout/retry rates, and validation-failure/filter-trigger/refusal rates, with alerting on anomalies.
- [ ] The highest-risk cases were verified — unvalidated output to a tool, schema failure with no path, unmeasured filter error rates, PII in prompts and logs, rendered refusals, and degenerate outputs persisted — not only the well-formed happy-path output.
