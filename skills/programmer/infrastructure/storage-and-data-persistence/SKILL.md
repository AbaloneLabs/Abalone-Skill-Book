# Storage And Data Persistence

Storage selection is the decision agents most often make by default — "give the database a volume, point the app at S3, done" — and it is the decision whose wrong choice is most expensive to reverse, because data accumulates on the chosen backend and moving terabytes or petabytes later is a migration project, not a reconfiguration. The central judgment problem is that storage is not one resource with a price; it is a family of technologies with radically different performance, durability, cost, access semantics, and operational characteristics, and the default that is convenient at launch becomes the wrong answer at scale. Block storage gives low latency and POSIX semantics but is tied to a single host and priced per provisioned IOPS; object storage is effectively infinite, multi-region, and cheap but has high per-request latency and eventual consistency on some operations; file storage shares nicely but is the hardest to scale and the most expensive per gigabyte. Choosing without understanding these tradeoffs produces a system that is either over-provisioned and costly, or under-provisioned and slow, or locked into a backend that cannot serve the access pattern that emerged.

The harm appears under the conditions real workloads hit: a database placed on standard block storage that exhausts its IOPS credits under load and stalls; a log/archive workload on premium SSD that costs ten times what object storage would; a stateful container whose persistent volume cannot be reattached after a node failure because the zone differs; a backup strategy that snapshots a volume but never tests restoration; a "cheap" storage tier whose retrieval latency breaks a user-facing read path. Agents tend to under-invest here because the cloud console offers a default and the happy path works. This skill is the discipline of matching the storage backend to the workload's access pattern, sizing it for the performance the workload demands, and managing the persistence lifecycle so data survives the failures and changes the system will experience. For backup and DR strategy see reliability/disaster-recovery-and-backup; for IaC volume definitions see terraform-and-iac-patterns; for database and data-model design see the data area. This skill is the storage backend and persistence layer itself.

## Core Rules

### Match The Storage Class To The Access Pattern, Not To Convenience

The first and most consequential decision is which class of storage the workload uses, because each class is optimized for a fundamentally different access pattern and is the wrong choice for the others. **Block storage** (EBS, managed disks) presents a raw block device with POSIX filesystem semantics, low single-digit-millisecond latency, and is attached to one host at a time — it is the right choice for a database, a local cache that must survive restart, or any workload needing random read/write with low latency, and the wrong choice for shared access or for large write-once archives. **Object storage** (S3, GCS, Blob) is a key-value store over HTTP with high per-request latency (tens of milliseconds), effectively unlimited capacity, multi-region durability built in, and a per-gigabyte price far below block — it is the right choice for blobs, backups, archives, static assets, data-lake objects, and anything write-once-read-many, and the wrong choice for random in-place updates or low-latency random reads. **File/shared storage** (NFS, EFS, FSx) allows multiple hosts to read and write the same filesystem concurrently, which is essential for some workloads (shared content, legacy apps expecting a filesystem) but is the most expensive per gigabyte and the hardest to scale performantly.

The judgment is to name the workload's dominant access pattern (random read/write low-latency, write-once-read-many large objects, shared concurrent access) and choose the class whose characteristics match, rather than defaulting to whatever the console suggests. A common and costly error is using block storage for an archive or backup workload that object storage would serve at a tenth of the cost; an equally common error is putting a latency-sensitive database on object or standard block storage and suffering stalls. The right class is the one whose performance, cost, and access semantics fit the workload — chosen deliberately, not inherited.

### Size For The Performance Characteristic The Workload Actually Demands, Not Just Capacity

Storage is not measured in gigabytes alone; it is measured in **IOPS** (input/output operations per second), **throughput** (MB/s), and **latency** (per-operation), and a volume sized only for capacity will fail under load if its performance characteristics are insufficient. This is the trap of provisioning "1 TB of storage" without asking how many IOPS that 1 TB delivers, because on many cloud block offerings the IOPS and throughput are tied to the volume size or type and are independent of how full the volume is. A database on a volume that delivers 3000 baseline IOPS will stall when a query workload demands 15000; the fix is a volume type or size that provisionally meets the demand, not more capacity.

The strong design sizes storage against the workload's measured or projected performance profile: peak IOPS, sustained throughput, and the latency the read path requires (a user-facing read tolerates single-digit milliseconds; a batch job tolerates hundreds). Provision headroom (workloads spike, and storage that runs at 100% utilization queues and stalls), understand whether the chosen volume type bursts (and what happens when the burst credit is exhausted — sustained stall), and separate the IOPS-bound workload from the throughput-bound one (many small random ops vs. few large sequential ones need different volume types). Monitor actual IOPS, throughput, queue depth, and latency against the provisioned limits, and alert before the workload hits the ceiling. A weak design discovers the IOPS limit during an outage; a strong design provisions and monitors against it.

### Design Volume Lifecycle And Stateful Workload Persistence Deliberately

Stateful workloads — databases, message brokers, caches that persist, any service whose disk state must survive a restart — require a persistence lifecycle that ephemeral containers and VMs do not, and the failure modes cluster around the attach/detach, migration, and failure semantics of persistent volumes. The hazards: a volume bound to a specific availability zone cannot be reattached in another zone after a node failure (the pod is rescheduled to a node in a different zone and cannot find its data); a volume attached to one node cannot be attached to another simultaneously (multi-attach is limited and often read-only); a volume resized online may require downtime or a specific filesystem resize; a snapshot taken without filesystem consistency (a crash-consistent snapshot of a running database) may be corrupt on restore.

The strong design makes the persistence lifecycle explicit: pin the workload's zone or use a multi-zone-capable volume type; understand whether the volume can fail over to another node and how fast; snapshot with application-level consistency (quiesce or use a coordinated snapshot, not a blind disk snapshot of a running database); test that a volume detached and reattached (or restored from snapshot) actually serves the workload; and plan for resize before the volume fills (monitor capacity, alert early, and know whether resize is online or requires downtime). For containerized stateful workloads, prefer a StatefulSet with stable volume claims over a Deployment with a shared volume, and understand that the volume's zone and the pod's scheduling constraints are coupled. A weak design treats persistent volumes as "a disk that follows the pod" and discovers the zone, multi-attach, and consistency limits during a failure; a strong design designs the lifecycle for the failures that will happen.

### Use Tiering And Lifecycle Policies To Match Cost To Access Frequency

Storage cost scales with volume and with the tier chosen, and the default of putting everything on the hot, expensive tier is the most common storage cost error — a workload that generates terabytes of logs or backups and leaves them on premium SSD or standard object storage forever accumulates a bill that dwarfs the compute. The discipline is to classify data by access frequency and move cold data to cheaper tiers automatically. **Object storage** offers lifecycle policies that transition objects to infrequent-access, then archive/glacier tiers after N days, and expire them after M days — a policy that should be the default for any object workload that retains data long-term. **Block storage** offers snapshot tiers and the option to stop paying for an idle volume by snapshotting and deleting it, then restoring on demand.

The judgment is to know the data's access pattern over time (hot for a week, warm for a quarter, cold thereafter, deletable after a year) and encode that as a lifecycle policy rather than leaving data on the expensive tier indefinitely. Beware the tradeoffs: archive tiers have retrieval latency of minutes to hours and retrieval fees, so a "cheap" tier is only cheap if the data is genuinely rarely accessed; tiering a workload whose data is accessed frequently to archive destroys the savings in retrieval costs and latency. Monitor storage spend by tier and by workload, and alert on anomalous growth — unbounded storage growth is a budget risk that is invisible until the bill arrives. A weak design puts everything on one tier and pays for capacity it never reads; a strong design matches tier to access frequency and automates the transitions.

### Treat Durability And Availability As Separate Properties And Verify Both

Durability (the probability that data is not lost) and availability (whether the data can be accessed right now) are different properties, and conflating them produces designs that are strong in one and absent in the other. Object storage typically offers 11-nines durability (data replicated across multiple facilities) but a single object's availability can drop during a regional event; block storage in a single zone has high availability within that zone but zero availability and potential data loss if the zone fails entirely. A workload that needs both high durability and high availability must design for them separately: replicate across zones or regions for availability, and rely on the backend's internal replication plus snapshots/backups for durability.

The strong design states the durability and availability requirements explicitly, chooses a backend and topology that meets each, and verifies by testing: can the data survive a zone failure (is it replicated or backed up cross-zone)? Can it survive accidental deletion or corruption (are there point-in-time snapshots or versioning)? Is the restore path proven (a snapshot never restored is a hope, not a backup — see reliability/disaster-recovery-and-backup)? A weak design assumes "the cloud stores it durably" and discovers, on the first zone failure or accidental deletion, that durability and availability were neither designed nor verified. Encryption (at rest, with keys managed independently — see configuration-and-secrets-management) is part of durability against the threat of theft or ransomware, not an afterthought.

## Common Traps

### Block Storage For An Archive Or Backup Workload

Putting logs, backups, or large write-once archives on premium block storage when object storage would serve the access pattern at a tenth of the cost, accumulating an oversized bill. Match the storage class to the access pattern; use object storage for write-once-read-many large objects.

### Sizing For Capacity And Ignoring IOPS And Throughput

Provisioning "1 TB" without checking the IOPS and throughput it delivers, then suffering stalls when the workload's random-read demand exceeds the volume's baseline and burst credits are exhausted. Size storage against measured IOPS, throughput, and latency with headroom, and monitor utilization against the limits.

### A Persistent Volume That Cannot Be Reattached After A Zone Failure

Binding a stateful workload's volume to one availability zone, then discovering after a node failure that the rescheduled pod lands in a different zone and cannot access its data. Pin the workload's zone or use a multi-zone-capable volume type, and understand the attach/detach and failover semantics.

### Crash-Consistent Snapshots Of A Running Database

Taking a disk snapshot of a database that is actively writing, producing a snapshot that may be corrupt or inconsistent on restore. Snapshot with application-level consistency (quiesce, or a coordinated snapshot), and verify by restoring and reading the data.

### Everything On The Hot Tier With No Lifecycle Policy

Leaving all data on premium or standard storage indefinitely with no tiering or expiry, so cold data accumulates a bill that dwarfs compute. Classify data by access frequency and encode transitions to cheaper tiers and expiry as lifecycle policies.

### Assuming "The Cloud Stores It Durably" Without Verifying Restore

Trusting the backend's durability marketing without testing that data survives a zone failure, an accidental deletion, or a corruption — discovering at restore time that the snapshot was never tested, the key is lost, or the cross-zone replica was never set up. State durability and availability separately, design for both, and verify by restoring.

## Self-Check

- [ ] The storage class (block, object, file/shared) was chosen to match the workload's dominant access pattern (random low-latency read/write, write-once-read-many large objects, shared concurrent access), not inherited as a console default — and block is not used for archives/backups that object storage would serve far more cheaply.
- [ ] Storage is sized for the performance the workload demands — IOPS, throughput, and latency — with headroom for spikes, the burst-credit behavior of the chosen volume type is understood, and actual utilization is monitored against provisioned limits with alerting before the ceiling is hit.
- [ ] The persistent-volume lifecycle for stateful workloads is designed for failure: the volume's zone coupling and multi-attach limits are understood, failover to another node is planned and tested, snapshots are application-consistent (not blind crash-consistent snapshots of a running database), and resize is planned before the volume fills.
- [ ] Storage cost is managed through tiering and lifecycle policies: data is classified by access frequency over time, cold data transitions to cheaper tiers automatically, expiry removes deletable data, and the retrieval latency and fees of archive tiers are acceptable for the data's real access frequency.
- [ ] Durability and availability are treated as separate properties, both stated explicitly: cross-zone or cross-region replication provides availability, the backend's internal replication plus snapshots/versioning provides durability, and encryption-at-rest uses independently managed keys.
- [ ] The restore path is proven — snapshots have been restored into a fresh environment and read, not merely configured — and the design survives a zone failure, an accidental deletion, and a corruption scenario (see reliability/disaster-recovery-and-backup).
- [ ] Containerized stateful workloads use a StatefulSet with stable volume claims (not a Deployment with a shared volume where appropriate), the pod scheduling constraints and volume zone are coupled deliberately, and the volume's attach/detach and failover behavior is verified.
- [ ] Storage spend is monitored by tier and by workload with alerting on anomalous growth, so unbounded retention does not silently accumulate a budget-breaking bill.
