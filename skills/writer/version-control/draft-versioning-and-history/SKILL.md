# Draft Versioning And History

Draft versioning is the discipline of keeping the history of a piece intact so that no good work is permanently lost and no bad revision is irreversible. Writers who do not version their drafts operate without a safety net: a revision that makes things worse cannot be undone, a cut passage cannot be recovered, and the question of which file is current becomes a source of anxiety and error. Versioning is not bureaucracy; it is the infrastructure that makes bold revision safe.

Agents tend to treat versioning as an afterthought, saving over files, using vague names like "final" and "final2," or relying on memory to track what changed. This fails exactly when it matters most, under deadline pressure or after a revision goes wrong, when recovery needs to be fast and certain. Use this skill when keeping prior versions, naming and organizing drafts, or maintaining a version history that lets you recover safely. The goal is a system that preserves good work, makes bad revisions reversible, and removes doubt about which version is current.

## Core Rules

### Never Save Over The Only Copy Of Good Work

The foundational rule of versioning is that any version worth keeping must exist as a separate, recoverable file. Saving over the previous draft destroys it permanently. Before any significant revision, preserve the current state.

Practices that protect work:

- save a new version before a major revision;
- keep the pre-revision draft intact and recoverable;
- never edit the only copy of a draft in place;
- treat deletion as a last resort, with material moved aside instead;
- assume any revision could make things worse and plan accordingly.

The cost of preserving a version is negligible. The cost of losing good work is not.

### Use A Naming Convention That Conveys Order And State

Vague names like "draft," "final," "v2," or "new" create confusion within days. A naming convention should make the order and state of each version unambiguous at a glance.

A workable convention includes:

- a stable base name for the piece;
- a version number or date that sorts chronologically;
- an optional state label such as draft, revised, edited, or final;
- avoidance of misleading terms like "final" before the piece is actually final;
- consistency across the whole project.

For example, a date-stamped scheme like `essay-2026-07-03-revised` sorts naturally and conveys both order and state. Choose one convention and apply it uniformly.

### Version At Meaningful Milestones, Not Continuously

Versioning every keystroke creates noise; versioning too rarely loses important states. Version at points where the draft has changed in a way you might want to return to.

Good milestone points:

- before a structural revision;
- after completing a full draft;
- before incorporating a round of feedback;
- before a major cut or rewrite;
- before sending to an editor or reviewer;
- at the end of a working session with substantial change.

The aim is enough granularity to recover any state that matters, without drowning in near-identical copies.

### Keep A Record Of What Changed Between Versions

A file name tells you when a version was saved but not why. A brief change log, even a simple one, makes the history useful rather than mysterious.

Record for each version:

- the date or version label;
- a one-line summary of what changed;
- whether the change was an improvement, a response to feedback, or an experiment;
- any known problems introduced or resolved.

A change log turns a pile of files into a navigable history and saves enormous time when you need to find the version where a particular passage was cut.

### Maintain A Single Clear Current Version

Version confusion often centers on which file is the current one. Multiple candidates with similar names lead to editing the wrong draft and losing work.

Establish clarity by:

- designating one file or location as the current working version;
- moving older versions to an archive folder or clear naming suffix;
- never leaving two files with ambiguous current status;
- communicating the current version to collaborators explicitly.

Anyone opening the project should be able to tell immediately which version is active.

### Preserve Cut Material In A Recoverable Place

Cuts are part of revision, but cut material is often useful later, in this piece or another. Deleting it permanently is rarely necessary and frequently regretted.

Instead of deleting:

- move cut passages to a notes or cuts file for the piece;
- keep them associated with the version they came from;
- retain them until the piece is truly final;
- review the cuts file before final lock in case something belongs back.

A cuts file makes bold cutting safe, because nothing is truly lost.

### Separate Drafting Versions From Delivered Versions

The versions you produce while working are different from the versions you deliver to editors, reviewers, or publication. Mixing them creates confusion about what was sent and what is current.

Keep separate:

- working drafts, which may be rough and experimental;
- delivered versions, sent to specific reviewers or editors;
- the canonical current version, which integrates returned feedback;
- the final locked version, once approved.

Tracking what was delivered to whom prevents the embarrassment of sending the wrong file or losing track of which feedback applies to which version.

### Back Up Beyond The Local Working Copy

A versioning system on a single device is vulnerable to device failure, accidental deletion, or sync errors. Backups protect the entire history, not just the current file.

Ensure:

- versions are stored in more than one location;
- cloud sync or backup covers the working directory;
- critical milestones are backed up deliberately;
- the backup is verified to be restorable, not assumed.

A backup you have never tested is a hope, not a safeguard.

### Plan Recovery Before You Need It

The value of versioning is realized at recovery time, often under stress. A system that is theoretically recoverable but practically hard to use fails when it matters.

Make recovery practical by:

- knowing how to find and restore a prior version quickly;
- keeping the history organized enough to locate a specific state;
- testing the recovery process once before relying on it;
- documenting where versions live and how they are named.

Recovery should be a calm, routine action, not a crisis investigation.

## Common Traps

### Saving Over The Previous Draft

Editing the only copy in place destroys recoverable work permanently. Always preserve before revising.

### Misleading Names Like Final And Final2

"Final" before the piece is final, and incrementing "final" labels, create confusion. Use dates or clear version numbers instead.

### Versioning Too Rarely Or Too Often

Too few versions lose important states; too many drown the history in noise. Version at meaningful milestones.

### No Record Of What Changed

Files without change notes become a mystery pile. A brief log makes history navigable.

### Ambiguity About Which Version Is Current

Multiple candidate files lead to editing the wrong draft. Designate and communicate the current version clearly.

### Permanently Deleting Cut Material

Cuts often belong back later. Move material aside rather than destroying it.

### Relying On A Single Device With No Backup

Device failure or sync errors can destroy the entire history. Back up beyond the local copy.

## Self-Check

Before treating the draft versioning system as sound, verify:

- No revision is ever saved over the only copy of good work.
- A naming convention conveys order and state unambiguously.
- Versions are saved at meaningful milestones, not continuously or rarely.
- A change log records what each version changed and why.
- One clear current version is designated and communicated.
- Cut material is preserved in a recoverable place rather than deleted.
- Working drafts, delivered versions, and the final version are kept separate.
- Backups exist beyond the local working copy and have been tested; the recovery process is practical and documented, not just theoretical
- Older versions are archived clearly so they do not create current-version confusion; the history is organized enough to locate any specific past state quickly
- Collaborators know which version is current and where versions live; the system makes bold revision safe rather than risky
