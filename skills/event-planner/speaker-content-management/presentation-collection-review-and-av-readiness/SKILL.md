# Presentation Collection Review And AV Readiness

Presentation management is where content quality, AV execution, legal risk, and schedule control meet. A speaker's deck may look fine on their laptop and fail on the event system because of fonts, aspect ratio, embedded video, permissions, missing adapters, unreadable text, inaccessible design, or late edits. Agents often say "collect slides in advance" without defining what to check, who approves content, how versions are controlled, and what happens when a speaker arrives with a new file. This skill helps manage presentation assets as production-critical materials.

## Core Rules

### Set Submission Requirements Before Speakers Build Decks

Speakers need clear instructions early: deadline, format, aspect ratio, file type, template, branding rules, maximum length, video specs, font rules, animation limits, naming convention, accessibility expectations, disclosure requirements, commercial limits, and upload method. If these are provided late, every deck becomes an exception.

Do not ask for "slides" generically. Specify PowerPoint, Keynote, PDF, Google Slides export, video codec, resolution, audio needs, and whether the event laptop or speaker laptop will be used. Include contact path for technical questions.

### Review Content And Technical Fitness Separately

Content review asks whether the deck matches the session objective, audience level, sponsor boundaries, legal requirements, claims, confidentiality, accessibility, and time. Technical review asks whether the file opens, displays correctly, uses the right aspect ratio, embeds media, includes fonts, plays audio, supports clicker control, and works on the show machine.

Both reviews matter and may have different owners. A technically valid deck can be a sales pitch that violates program rules. A strong educational deck can fail if the video is linked to a local file. Track each review status explicitly.

### Control Versioning And File Naming

Late versions are unavoidable, but uncontrolled versions create show risk. Use consistent naming with event, session, speaker, date, version, and status. Store files in a single source of truth. Decide when decks freeze, who can approve exceptions, and how final files reach the show machine.

Avoid running from email attachments or random USB drives unless there is a deliberate emergency process. If a deck changes after rehearsal, mark it clearly and retest affected elements. Old versions should be archived or removed from the operating folder to prevent accidental use.

### Test On The Actual Playback Environment

Presentations should be tested on the actual or equivalent show system, not only on the speaker's laptop. Check aspect ratio, fonts, videos, audio routing, animations, clicker, confidence monitor, presenter notes if used, captions, remote speaker sharing, screen resolution, and display scaling. Hybrid events add streaming platform and recording considerations.

If speakers use their own laptops, test adapters, output resolution, power, screen sleep, notifications, privacy, and network needs. Speaker laptops add variability and should be used only when necessary for demos, proprietary tools, or unusual media.

### Handle Embedded Media And Live Demos As High-Risk Segments

Videos, audio, web links, live demos, polls, browser tabs, and software walkthroughs fail more often than static slides. Require media files separately where possible, test playback offline, confirm sound routing, and have backup screenshots or recordings. Live internet-dependent demos need backup plans.

For product launches, technical conferences, or high-profile keynotes, rehearse risky segments with production. Decide who can abort a demo, switch to backup, or fill time. Do not let the first full test happen onstage.

### Check Readability And Accessibility

Slides that look good on a laptop may be unreadable in a ballroom or livestream. Consider font size, contrast, color dependence, alt text where distributed, captions for video, image rights, language level, and visual density. Remote viewers may see slides on small screens. Colorblind attendees may miss meaning encoded only by color.

Provide guidance to speakers and flag serious issues early. Accessibility is not just compliance; it improves comprehension. If slides will be shared afterward, accessibility and rights become more important.

### Manage Legal, Brand, And Confidentiality Review

Presentations can include customer logos, screenshots, financial claims, health claims, legal statements, unpublished product details, personal data, copyrighted media, stock images, medical claims, or sponsor promises. Events with regulated content need content review before publication or recording.

Define what requires legal, compliance, sponsor, client, or brand approval. Speakers should disclose third-party media, customer examples, data sources, and embargoed material. If recording or sharing decks, confirm release rights.

### Prepare Onsite Change And Emergency Process

Speakers may bring last-minute decks, corrected slides, new videos, or sponsor updates. The event needs an emergency intake process that protects the show: file virus scan if applicable, naming, technical test, approval, copy to show machine, backup copy, and communication to stage manager.

Set boundaries. A last-minute typo fix is different from replacing a keynote deck. If a change cannot be tested, the risk should be acknowledged and approved by the right owner.

## Common Traps

### Collecting Slides Without Specs

Late technical requirements force rework and create inconsistent decks.

### Assuming A Deck That Opens Is Ready

Content, media, fonts, timing, accessibility, and legal review still matter.

### Running From Email Attachments

Uncontrolled file sources increase the chance of using the wrong version.

### Testing On The Wrong Machine

Speaker laptops and show machines can render fonts, video, and aspect ratio differently.

### Ignoring Audio In Embedded Videos

Video may play silently if audio routing is not tested with AV.

### Letting Live Demos Depend On Venue Internet Alone

Internet, authentication, permissions, and software updates can fail live.

### Overlooking Rights And Confidentiality

Recorded or shared slides can expose copyrighted, customer, personal, or embargoed information.

### Accepting Untested Last-Minute Replacements

Late deck swaps need approval and a fast technical check, not blind trust.

## Self-Check

- [ ] Were slide requirements, deadline, format, aspect ratio, template, media specs, and naming rules sent early?
- [ ] Are content review and technical review tracked separately with owners?
- [ ] Is there a single source of truth with version naming, freeze rules, and archive of old files?
- [ ] Have decks been tested on the actual or equivalent playback environment?
- [ ] Are videos, audio, fonts, animations, clicker, confidence monitor, captions, and remote display checked?
- [ ] Are live demos, web links, polls, and software walkthroughs backed up or rehearsed?
- [ ] Are readability, contrast, font size, color dependence, and accessibility considered for room and remote viewers?
- [ ] Are legal, brand, sponsor, customer, data, copyright, and confidentiality issues reviewed where relevant?
- [ ] Is onsite change intake defined with approval, testing, copying, and communication steps?
- [ ] Could the show team confidently identify and play the correct final file without asking the speaker live?
