# Packaging And Scope To Price Fit

Packaging is where buyer need, product entitlement, delivery effort, and price meet. A poor package can oversell unused capability, underscope required services, hide implementation cost, or force discounting because the buyer cannot see fit. This skill helps the agent align package and scope to the buyer's actual problem and willingness to pay.

## Core Rules

### Start from required outcomes

Choose packaging based on the outcomes, workflows, users, integrations, data, support needs, and success criteria the buyer confirmed. Do not start from the package the seller wants to sell.

The right package is the one that can deliver value credibly.

### Separate must-have from nice-to-have

Identify which modules, features, services, support levels, and add-ons are essential for success versus optional. This helps avoid overpackaging and underpackaging.

Buyers should understand what drives price.

### Check entitlement and limits

Plan tiers may differ by seats, usage, storage, API access, integrations, security, admin controls, reporting, support, SLA, regions, and compliance. Make sure the selected package actually includes required capabilities.

Do not solve packaging gaps with vague promises.

### Include services and implementation

If value depends on onboarding, migration, integration, training, data cleanup, change management, or premium support, include those in scope and price. Leaving services out to lower price creates delivery risk.

### Fit package to buying motion and avoid discounting as packaging substitute

SMB, mid-market, enterprise, public sector, partner-led, and self-serve buyers may need different package structures, terms, and support. A package that fits product need may not fit procurement or operational reality.

If the buyer objects because the package includes unneeded scope, consider right-sizing before discounting. If the package lacks required capability, discount does not fix fit.

### Explain tradeoffs and check adoption capacity

When comparing packages, state tradeoffs in capability, support, scalability, risk, implementation effort, and future flexibility. Avoid presenting the highest tier as automatically best.

Tradeoffs should be tied to the buyer's own constraints. A lower tier may reduce cost but increase manual work, limit reporting, delay integrations, or leave security requirements unmet. A higher tier may improve scale and governance but require more change management and executive sponsorship. The point is not to make every buyer choose more; it is to make the consequences of each package visible.

Packaging should fit the buyer's ability to implement and adopt. A broad package can fail if the buyer lacks admin time, project leadership, training capacity, clean data, or executive sponsorship. Conversely, a package that is too narrow may fail because users never reach the workflow where value is created. Match scope to the buyer's real operating capacity.

### Avoid hidden cross-subsidies and make down-sell and up-sell logic explicit

Do not use one part of the package to silently subsidize another, such as reducing services below required effort to preserve subscription price, or overloading unused modules to justify a discount. Hidden cross-subsidies create delivery pressure and make future renewals hard to explain.

If recommending a smaller start, explain why it is enough for the first outcome and what it will not solve. If recommending a larger package, explain what risk or value justifies the added scope. This protects the recommendation from looking like arbitrary price steering.

### Validate package language against entitlements and document accepted limitations

Buyer-facing package names often sound broader than their actual entitlements. Confirm what each term means in the price book, product catalog, support policy, and contract. Do not let a friendly package description become an entitlement promise.

If the buyer chooses a lower tier or reduced scope, document what they are not getting and what risks remain. Post-sale teams need this context.

### Watch for future expansion path and align contract and handoff

If starting smaller, define what would trigger expansion: users, volume, integrations, support needs, security, reporting, or new departments. This makes land-and-expand honest.

Package, scope, price, contract terms, order form, CRM, and implementation handoff should match.

## Common Traps

- Selling the preferred package before confirming required outcomes.
- Bundling nice-to-have features as if required; missing tier limits around API, security, reporting, support, or regions
- Excluding implementation services to make price look better; using discount to hide wrong packaging
- Ignoring procurement or segment buying motion; failing to explain package tradeoffs
- Letting the buyer choose lower scope without documenting risk; calling a small start "land and expand" without expansion triggers
- Having contract, CRM, and handoff disagree about package; overpackaging customers who will not adopt
- Underpackaging customers who need support to succeed; explaining package tradeoffs in seller terms rather than buyer consequences
- Ignoring whether the buyer can actually adopt the proposed breadth; hiding required service effort to keep the package price low; making a down-sell or up-sell recommendation without rationale; assuming package names clearly describe contractual entitlements

## Self-Check

- Is packaging based on confirmed outcomes and workflows?
- Are must-haves separated from nice-to-haves?
- Are tier limits and entitlements checked?
- Are seats, usage, storage, integrations, API, security, reporting, support, SLA, region, and compliance needs covered?; are services and implementation included where required for value?
- Does the package fit the buyer's segment and buying motion?; was right-sizing considered before discounting?
- Are tradeoffs explained clearly?; are accepted limitations documented?
- Are expansion triggers defined for smaller starts?; do contract, order form, CRM, and handoff match?
- Is the package likely to produce post-sale success?; are tradeoffs expressed as buyer consequences?
- Does proposed scope fit buyer adoption capacity?; are services, subscription, and discount economics transparent rather than cross-subsidized?
- Is the logic for smaller or larger scope explicit?; do package names and descriptions match actual entitlements?
