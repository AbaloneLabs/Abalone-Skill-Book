# Implementation Readiness Before Sale

Implementation readiness is a sales issue, not only a post-sale issue. A deal can be properly contracted and still fail because the buyer lacks owners, data, technical access, time, or change capacity. Agents often postpone readiness questions to avoid slowing the sale, but that creates churn and trust risk. This skill helps the agent inspect readiness before promises become obligations.

## Core Rules

### Identify implementation owners early

Name buyer-side project owner, executive sponsor, technical owner, admin, data owner, training owner, and decision maker for scope changes. If the buyer cannot identify owners before sale, kickoff may begin with confusion.

### Assess data and system readiness

Ask what data, systems, integrations, permissions, environments, APIs, exports, identity providers, and admin access are required. Check whether data is clean, available, owned, and legally usable. Data readiness issues often drive timeline slips.

### Validate integration feasibility and check project governance

Confirm required integrations, technical constraints, authentication, security review, legacy systems, vendor dependencies, and unsupported edge cases. Do not treat "we have an API" as proof that the buyer's implementation will work.

Implementation needs a decision structure. Confirm who approves scope changes, resolves blockers, accepts milestones, handles escalation, and communicates internally. Without governance, even simple implementations can stall when decisions are required.

### Assess change management maturity and define acceptance criteria

Ask how the buyer introduces new workflows, trains users, enforces adoption, handles resistance, and measures behavior change. A product can be configured correctly and still fail if the buyer has no change plan. Readiness includes organizational capacity, not only technical access.

Clarify what the buyer will consider successful implementation: data loaded, users trained, workflow live, integration tested, first report delivered, compliance signoff, or value milestone achieved. Vague launch expectations create disputes about whether implementation is complete.

### Align services with readiness gaps and check contract-to-implementation continuity

If the buyer lacks capacity, determine whether included services are enough or whether paid services, partner help, premium onboarding, or phased implementation is required. Do not hide needed services to keep the commercial offer attractive.

Make sure contract terms, SOW language, implementation package, start date, support tier, and buyer responsibilities match the readiness assessment. If the contract implies one implementation path while the readiness review assumes another, the gap will surface after signature.

### Decide whether validation is needed before sale and test timeline realism

For material uncertainty, consider technical workshop, data sample review, proof of concept, admin readiness check, or implementation planning session before final commitment. Validation should be proportional to risk; it is not bureaucracy when it prevents an unrealistic sale.

Compare desired go-live date with procurement, legal, security, data prep, configuration, training, testing, and buyer decision cycles. If the timeline is aggressive, define what must be true and what scope may need to shrink.

### Clarify buyer responsibilities and align scope with readiness

Implementation success depends on buyer actions: providing access, making decisions, attending training, cleaning data, assigning users, communicating change, and reviewing milestones. Put these responsibilities in proposal, MAP, and handoff.

If readiness is limited, recommend phased rollout, narrower scope, pilot, assisted services, delayed start, or prerequisite work. Do not sell broad scope that the buyer cannot absorb.

### Surface readiness risks to delivery and tie readiness to value realization

Implementation, customer success, support, security, and product teams should see readiness risks before close. If they need to approve special effort or timeline, get acceptance before contracting.

Readiness is not only about launch. Data quality, adoption owner, training, and process change affect whether promised value is measurable. Include readiness in the value case.

### Avoid reassurance without plan and document readiness state

"Our team will help" is not enough. State what help is included, what is not included, what the buyer must do, and how risks will be managed.

Record confirmed readiness, unknowns, risks, buyer responsibilities, seller responsibilities, dependencies, and mitigation in CRM and handoff artifacts.

## Common Traps

- Treating implementation readiness as post-sale discovery; proceeding without named buyer owners
- Ignoring data access, quality, and ownership; assuming API availability means integration feasibility
- Accepting buyer go-live dates without critical path review; hiding buyer responsibilities until kickoff
- Selling broad scope to a buyer with limited capacity; failing to involve delivery before accepting special timelines
- Separating readiness from value realization; reassuring buyers without a concrete implementation plan
- Leaving readiness risks out of CRM and handoff; letting seller urgency override implementation reality
- Ignoring implementation governance and escalation ownership; treating change management as outside the sale
- Starting without buyer acceptance criteria; excluding needed services to make the price easier; letting contract terms and readiness assumptions diverge; skipping pre-sale validation when uncertainty is material

## Self-Check

- Are buyer implementation owners identified?
- Are data, system, permission, and access requirements understood?
- Is integration feasibility validated beyond generic capability?; is timeline realistic against critical path?
- Are buyer responsibilities explicit?; is scope matched to buyer capacity?
- Have delivery and post-sale owners seen readiness risks?; is readiness tied to value realization and success metrics?
- Are reassurances backed by included services and plans?; are readiness risks, unknowns, and mitigations documented?
- Would kickoff be able to start from the current information?; should scope, timing, or qualification change before sale?
- Is project governance defined?; is change management maturity assessed?
- Are implementation acceptance criteria explicit?; are readiness gaps matched to services, partner help, or phased scope?
- Do contract, SOW, support tier, and implementation assumptions match readiness?; is pre-sale validation planned where uncertainty is material?
