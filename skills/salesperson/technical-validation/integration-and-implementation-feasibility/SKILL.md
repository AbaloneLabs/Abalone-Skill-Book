# Integration And Implementation Feasibility

Implementation feasibility determines whether a sold deal can become a successful customer. Sales conversations often focus on whether the product can technically support a use case, but the harder question is whether the buyer has the systems, people, data, time, and governance to implement it. This skill helps the agent test feasibility before promises become delivery obligations.

## Core Rules

### Identify the implementation path

Clarify whether implementation is self-serve, guided onboarding, professional services, partner-led, technical integration, migration, phased rollout, or custom project. Each path has different cost, timeline, and ownership.

Do not let the buyer assume a high-touch implementation if it is not included.

### Map buyer responsibilities

Identify who on the buyer side owns admin setup, data preparation, integration, security approval, user training, change management, testing, and go-live. If no owner exists, implementation risk is high.

Buyer resources are part of feasibility.

### Assess data readiness

Check data quality, format, source systems, duplicates, missing fields, permissions, migration volume, historical records, cleanup work, and who can approve data changes. Poor data can derail even simple implementations.

Do not promise migration ease before data is understood.

### Validate integration constraints

Review APIs, connectors, authentication, rate limits, data direction, sync frequency, error handling, sandbox access, monitoring, and third-party ownership. Some integrations are technically possible but operationally fragile.

Integration feasibility includes maintenance.

### Surface timeline realism

Compare desired go-live date with security review, procurement, data prep, technical work, testing, training, holidays, and stakeholder availability. Buyers often underestimate elapsed time.

Do not compress implementation timelines to close the deal.

### Separate standard from custom

State what is standard, what is configuration, what is services, what is partner work, what is custom, and what is not supported. Custom assumptions must be approved before proposal.

Ambiguity becomes scope creep.

### Identify adoption risk and create handoff-ready implementation notes

Implementation is not complete when settings are configured. Consider training, user readiness, workflow change, internal communications, support model, and success metrics.

Adoption risk should influence deal qualification.

Record scope, assumptions, dependencies, buyer owners, technical requirements, risks, exclusions, timeline, and promises. Delivery teams should not rediscover what sales already learned.

### Validate environment and access assumptions and identify cutover and rollback needs

Implementation may depend on sandbox access, admin permissions, test users, API credentials, data export rights, firewall rules, procurement approval, or vendor cooperation. Confirm who can provide each access and how long it normally takes.

Access delays are often mistaken for product delays. Capture them before promising go-live dates.

For workflows that affect production, decide whether the buyer needs phased rollout, parallel run, migration backup, rollback plan, maintenance window, or user communications. A technically feasible integration can still fail if cutover is unmanaged.

Sales should not sell a launch path that implementation cannot safely execute.

### Align feasibility with commercial scope and confirm post-sale owner acceptance

If implementation requires services hours, partner support, premium onboarding, custom work, or buyer professional services, ensure the commercial proposal includes it. Do not leave required work unfunded or implied.

Unfunded implementation assumptions become customer disappointment.

When feasibility depends on implementation, customer success, support, partner, or services capacity, confirm that the post-sale owner accepts the assumptions before close. Sales confidence is not enough.

If the post-sale owner has not accepted the plan, mark the deal as carrying delivery risk.

## Common Traps

- Treating implementation as a generic post-sale problem.
- Assuming buyer resources exist because the buyer is enthusiastic.
- Ignoring data cleanup and migration complexity; saying an integration exists without checking direction, limits, ownership, and monitoring
- Accepting buyer go-live date without validating dependencies; blurring standard configuration, services, partner work, custom work, and unsupported work
- Forgetting user adoption and change management; hiding implementation risk to avoid slowing the deal
- Leaving implementation notes in seller memory instead of handoff records; promising timelines or services that delivery has not accepted
- Assuming sandbox, admin, API, test user, firewall, or vendor access will be available immediately; ignoring cutover, rollback, backup, and user communication needs for production workflows
- Selling a commercial package that omits services or partner work required for success; treating a sales feasibility judgment as accepted delivery commitment without post-sale owner review

## Self-Check

- Is the implementation path clear?
- Are buyer owners identified for setup, data, integration, security, training, testing, and go-live?
- Is data readiness assessed?
- Are integration direction, API, connector, authentication, rate limit, sandbox, monitoring, and third-party ownership checked?
- Is timeline realistic against review, procurement, data prep, testing, training, and availability?
- Are standard, configuration, services, partner, custom, and unsupported work separated?
- Is adoption risk assessed beyond technical launch?; are delivery assumptions and exclusions approved where needed?
- Are handoff notes complete enough for implementation teams?; does feasibility affect qualification, scope, and forecast?
- Are environment access owners and lead times confirmed?; are cutover, rollback, backup, phased rollout, and user communication needs identified?
- Does the commercial scope include required services, partner work, and buyer responsibilities?; has the responsible post-sale owner accepted the feasibility assumptions or marked residual delivery risk?
