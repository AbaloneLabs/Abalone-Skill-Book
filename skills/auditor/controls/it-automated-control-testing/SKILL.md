# IT-Dependent And Automated Control Testing

Automated controls are controls performed entirely or substantially by a system without manual intervention: a three-way match that blocks payment, an access rule that prevents conflicting privileges, a validation that rejects out-of-range entries. Because they execute consistently every time, a single effective test can, in principle, demonstrate operation across the whole period, which is why automation is so attractive for controls-reliance. But this attractive property is also where the failures cluster. An automated control is only as reliable as the configuration that defines it, the data it processes, and the IT general controls that prevent unauthorized change to it. Agents frequently test the automated control once, conclude it operates throughout the period, and ignore whether someone could have changed the configuration, whether the underlying data was complete and accurate, or whether the control was even active for the full period. That produces a controls-reliance conclusion resting on a single point of failure.

Use this skill when testing automated or IT-dependent controls, when deciding whether to rely on them across a period, when assessing the IT general controls that support them, and when system changes complicate the period-long reliability of a control. The goal is to test automated controls in a way that genuinely supports period-long reliance, including the configuration, the data, and the controls over change.

## Core Rules

### Understand What Makes An Automated Control Reliable

An automated control's reliability comes from its consistency, but that consistency depends on conditions that must each hold. Reliability requires that the control was correctly configured, that it operated for the entire period, that the data it processed was complete and accurate, and that no unauthorized change altered its behavior.

Confirm reliability conditions:

- the control logic is correctly designed to address the risk;
- the configuration was active and unchanged for the relevant period, or changes were controlled;
- the data sources the control uses are complete, accurate, and authorized;
- the system generating the control has not been compromised or bypassed;
- exceptions generated by the control are investigated and resolved.

Testing the control once without confirming these conditions demonstrates only that it worked at one point, not that it operated reliably across the period.

### Test The Control Logic And Configuration, Not Just The Output

The output of an automated control, such as a rejection or an exception report, is the visible end of a configuration. The reliability of the control depends on the configuration behind the output. Testing must examine the logic, not only confirm that the output appeared.

Test configuration by:

- inspecting the actual system configuration, not the documented standard;
- confirming the parameters, thresholds, and rules match the control objective;
- testing the control with data designed to trigger and to bypass it, to confirm it behaves as intended;
- verifying that the control cannot be overridden by users who should not have that ability;
- confirming the control covers the entire relevant population, with no excluded segments.

A control that appears to operate but has mis-set thresholds or excludes part of the population provides false assurance.

### Assess The IT General Controls That Support The Automated Control

Automated controls depend on IT general controls, particularly change management and access controls. If someone can change the configuration without authorization or testing, or if excessive access allows bypass, the automated control's period-long reliability is undermined regardless of how well it tested at a point in time.

Assess supporting general controls:

- change management: are changes to the application and its configuration authorized, tested, and approved before production;
- access controls: is access to configure, override, or bypass the control restricted and monitored;
- computer operations: are processing failures, errors, and re-runs detected and resolved;
- segregation between development, testing, and production environments.

Where IT general controls are weak, an automated control cannot be relied upon across the period without additional testing of whether the configuration remained intact.

### Test The Completeness And Accuracy Of The Data The Control Uses

An automated control operates on data. If the data is incomplete or inaccurate, the control may execute correctly on bad data and still fail to address the risk. The completeness and accuracy of reports and data used by the control must be tested.

Test data reliability by:

- reconciling the data source to the authoritative system or control total;
- confirming the population includes all relevant transactions, with no silent exclusions;
- testing key fields for accuracy and completeness;
- verifying that interfaces feeding the data operate correctly;
- confirming the data has not been altered after the fact.

A control that matches invoices to purchase orders reliably but operates on a population missing manual or off-system transactions does not cover the risk.

### Determine Whether A Single Test Can Support Period-Long Reliance

The efficiency of automated controls is that a single effective test can, in principle, support reliance across the period, because the control executes identically each time. But this is only valid where the configuration was unchanged and the supporting general controls are effective. Where changes occurred, the period must be segmented and the control tested for each configuration version.

Determine the testing approach by:

- confirming through change logs whether the configuration was stable across the period;
- where stable and general controls are effective, a single test of the current configuration may suffice, supported by general control testing;
- where changes occurred, identifying each configuration version and testing each, or testing at points that bracket the changes;
- documenting the basis for the testing approach and its reliance on general controls.

Assuming single-test reliability without confirming configuration stability and general control effectiveness is the most common error.

### Handle Manual Components And IT-Dependent Manual Controls

Many controls are neither fully automated nor fully manual, but IT-dependent: a system generates a report or exception list, and a person reviews and acts on it. These controls combine the reliability concerns of both automated and manual controls.

For IT-dependent manual controls, test:

- the automated component: is the report or exception list complete, accurate, and timely;
- the manual component: did the reviewer actually examine it, investigate exceptions, and act;
- the interaction: does the manual review address what the automated component surfaces.

A strong automated report reviewed by a signer who does not investigate is not an effective control, regardless of the system's reliability.

### Consider Bypass, Override, And Workaround Pathways

Automated controls can be bypassed through manual workarounds, administrator access, or processes that operate outside the system. The assessment should consider whether the control can be circumvented, not only whether it operates when used as intended.

Investigate bypass pathways:

- whether manual journal entries or off-system processes bypass the automated control;
- whether administrator or super-user access allows override without detection;
- whether users have developed workarounds for controls they find inconvenient;
- whether exceptions are routinely overridden rather than investigated.

A control that operates perfectly within the system but is routinely bypassed in practice provides no real protection.

## Common Traps

### Testing Once And Concluding Period-Long Reliability

A single test demonstrates operation at a point in time. Period-long reliance requires confirming configuration stability and general control effectiveness.

### Ignoring IT General Controls

Automated controls depend on change management and access controls. Weak general controls undermine the reliability of every automated control that depends on the system.

### Accepting The Output Without Testing The Configuration

Confirming that a control produced an output does not prove the configuration is correct. Inspect the logic and parameters.

### Overlooking Data Completeness And Accuracy

A control operating on incomplete or inaccurate data executes correctly but fails to address the risk. Test the data the control uses.

### Assuming Configuration Stability Without Verification

Without checking change logs, the auditor cannot know whether the configuration was stable. Changes require segmented testing.

### Missing Bypass And Workaround Pathways

A control that is routinely bypassed or overridden provides no protection, regardless of how well it operates when used as intended.

### Treating IT-Dependent Manual Controls As Fully Automated

A system-generated report reviewed manually is only as effective as the manual review. Test both components.

## Self-Check

- Is the automated control's reliability assessed against configuration correctness, period of operation, data completeness and accuracy, and protection from unauthorized change?
- Has the actual system configuration been inspected and tested with data designed to trigger and bypass the control, rather than only confirming the output?
- Have the supporting IT general controls, particularly change management and access, been assessed, and does their effectiveness support period-long reliance?
- Has the completeness and accuracy of the data the control uses been tested, including reconciliation to authoritative sources and interface integrity?
- Is the testing approach, single test or segmented, justified by confirmed configuration stability and general control effectiveness, with the basis documented?
- For IT-dependent manual controls, are both the automated report generation and the manual review and exception follow-up tested?
- Have bypass, override, administrator access, and manual workaround pathways been investigated, not only the control's operation when used as intended?
- Where configuration changes occurred, has the period been segmented and the control tested for each relevant version?
- Is the reliance on automated controls supported by evidence that the conditions for period-long reliability actually hold?
- Could an independent reviewer confirm that the controls-reliance conclusion rests on configuration, data, and general control testing, not on a single point-in-time test?
