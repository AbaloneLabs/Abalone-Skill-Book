# Project To Run State Transition

Project-to-run transition is where temporary delivery work becomes recurring operational responsibility. The risk is not only that something is unfinished; it is that no one has designed how the new work will be owned, funded, monitored, improved, and supported after the project team leaves. This skill helps the agent plan the transition from implementation mode to steady-state operations.

## Core Rules

### Define the target run state

Describe how the process, system, service, or capability will operate after transition: owner, users, customers, volume, service levels, support hours, tooling, reporting, controls, vendor role, staffing, funding, and governance cadence. A target run state should be concrete enough that a manager can operate it.

Do not let "business as usual" remain undefined. The new work may not fit the old business-as-usual model.

### Transfer ownership with authority and resources

Ownership transfer should include decision rights, budget, staffing, access, vendor authority, reporting responsibility, change authority, and escalation path. A named owner without authority or capacity cannot run the service.

Clarify which responsibilities remain with project, product, IT, vendor, finance, compliance, or operations during transition and after steady state begins.

### Convert temporary project artifacts into operating artifacts

Project plans, design documents, test notes, issue lists, and meeting decisions are not enough for run state. Operations needs runbooks, SOPs, queue categories, support scripts, monitoring dashboards, training materials, control evidence, knowledge articles, and service catalog entries.

Artifacts should be placed where operational teams actually work, not buried in project folders.

### Define steady-state governance

Recurring governance should cover performance, quality, incidents, change requests, defects, backlog, vendor issues, capacity, compliance, and improvement. Define cadence, attendees, inputs, decisions, and escalation.

Without governance, unresolved project issues become operational noise and nobody knows whether the new service is healthy.

### Reconcile residual risks and deferred work

Projects often close with deferred defects, manual workarounds, open risks, pending controls, incomplete reports, or future enhancements. Decide which items block transition, which enter hypercare, which belong in steady-state backlog, and which require formal risk acceptance.

Deferred work should have owner, priority, date, and impact. Do not let "phase two" become a place where operational risk disappears.

### Validate capacity and workload

Run-state work may add queue volume, monitoring tasks, manual reviews, reporting, vendor management, training, access requests, and support demand. Estimate workload by role, shift, skill, and location. Include seasonal and peak periods.

If capacity is not funded or staffed, either reduce scope, revise service levels, automate, or accept risk explicitly.

### Plan communication and expectation reset

Affected teams need to know what changed, who owns it, how to request help, what service level applies, what old paths are retired, and what limitations remain. Customers or employees may need updated instructions, templates, or status messages.

Communication should make the transition visible enough to prevent people from continuing old workarounds.

### Confirm project exit and support availability

Define when project resources roll off, what support they still provide, how questions are handled, and how long expert support remains available. If vendor or project experts leave immediately, operations may be unable to resolve early issues.

Project exit should depend on stabilization evidence, not only closeout date.

### Align funding with the run model

Run-state ownership needs recurring funding for licenses, support labor, vendors, maintenance, reporting, quality checks, training, and improvement. Confirm which costs move from project budget to operating budget and when.

If funding is not transferred, operations may keep the service alive by cutting corners elsewhere.

## Common Traps

- Calling something run state before the target operating model is defined.
- Assigning an owner without budget, staffing, access, or decision rights.
- Handing over project documents instead of operational artifacts.
- Leaving old request paths, forms, or workarounds active after transition.
- Treating residual defects and risks as harmless because the project is closing.
- Hiding unfunded workload inside existing teams; deferring control gaps or reporting gaps without formal risk acceptance
- Ending project support before operations can diagnose common failures; failing to define steady-state governance for quality, backlog, incidents, changes, and vendors
- Not telling users how to get service after the transition; moving project costs into operations without recurring budget, owner, or tradeoff decision

## Self-Check

- Is the target run state defined for owner, users, volume, service levels, support, tooling, reporting, controls, vendors, staffing, funding, and cadence?
- Does the owner have authority, budget, capacity, access, vendor leverage, and change rights?
- Are responsibilities separated for project, product, IT, vendor, finance, compliance, and operations before and after transition?
- Are project artifacts converted into runbooks, SOPs, queues, scripts, dashboards, training, controls, knowledge, and service catalog entries?
- Are operational artifacts placed where teams actually work?
- Is steady-state governance defined for performance, quality, incidents, changes, defects, backlog, vendors, capacity, compliance, and improvement?
- Are residual risks, deferred defects, workarounds, incomplete controls, and reports classified with owner, priority, date, impact, and acceptance?
- Is run-state workload estimated by role, shift, skill, location, season, and peak?
- Are old paths retired and new request, support, and service expectations communicated?
- Does project exit depend on stabilization evidence and available expert support?; are recurring costs and operating budget transfers defined for labor, licenses, vendors, maintenance, reporting, quality, training, and improvement?
