# Systematic And Reproducible Searching

Some searches are casual explorations; others are evidence. A search that feeds a systematic review, a clinical guideline, an environmental impact report, a grant proposal, or a formal literature update must be more than effective. It must be documented precisely enough that another qualified searcher could run it and get the same results, and that a reviewer could audit what was searched, when, and why. Reproducibility is what separates a defensible evidence base from an opinion backed by cherry-picked citations. The trap is that reproducibility is invisible at the moment of searching. A searcher who finds great results but did not record the exact query, the database version, the filters, and the date has produced work that cannot be verified, updated, or trusted at the level the project demands. Systematic searching is a discipline of documentation as much as of retrieval, and it must be planned before the search runs, not reconstructed afterward.

Use this skill when a search must serve as evidence (systematic and scoping reviews, guidelines, grants, policy briefs, formal literature updates), when writing a search methods section, and when teaching researchers to produce reproducible searches. The goal is to prevent the agent from running searches that cannot be reproduced, from under-documenting methods, from failing to report databases and dates, or from treating a systematic search like an ordinary reference lookup.

## Core Rules

### Plan The Search Protocol Before Running It

A reproducible search is designed before it is executed. Ad hoc searching cannot be reconstructed accurately. Define the protocol first.

A search protocol specifies:

- the review question, framed with a tool like PICO (population, intervention, comparator, outcome) where appropriate;
- the eligibility criteria: inclusion and exclusion rules for population, design, date range, language, publication type;
- the databases and sources to be searched, named explicitly;
- the search strategy: concepts, synonyms, controlled vocabulary, operators;
- limits and filters to be applied, and at what stage;
- how results will be managed: deduplication, screening, reporting.

A protocol makes the search auditable and forces decisions about scope before results bias them.

### Decompose The Question Into Concepts And Document Each

Reproducibility depends on a structured, documented strategy, not a natural-language phrase. Decompose the question and record the building blocks.

Document:

- each core concept and its synonyms, variants, and broader or narrower terms;
- the controlled vocabulary terms used per database (MeSH, Emtree, ERIC descriptors), with the explode and focus settings;
- how concepts are combined (AND across concepts, OR within a concept);
- the Boolean and proximity operators and nesting, exactly as entered;
- any limits (date, language, publication type, species, age group).

(See the related skill on search terms and controlled vocabulary for the mechanics of building the strategy.)

### Record The Exact Query Per Database

Each database interprets searches differently. The exact query string, in each database, must be recorded verbatim, because a strategy does not transfer cleanly between platforms.

For each database, record:

- the database name, platform, and version or interface (interfaces change and affect results);
- the full query string with all line breaks and operators;
- which lines were subject headings versus keywords;
- the filters or limits applied and whether they were applied within the query or post-search;
- the date and time the search was run;
- the number of results retrieved.

A screenshot or exported file of the search history is the most reliable record. Hand-transcribed queries introduce errors.

### Report Sources, Dates, And Limits Transparently

A methods section that says "we searched PubMed and a few other databases" is not reproducible. Transparency is the standard.

Report:

- every source searched, including gray literature and trial registries, not just the main databases;
- the date range searched and the date the searches were run;
- any language or publication-type restrictions and their justification;
- hand-searching of key journals and reference lists;
- contacts with authors or experts;
- the number of records retrieved from each source.

Anyone reading the methods should be able to judge whether the search was adequate and could repeat it.

### Use Peer Review Of The Search Strategy

A single searcher's strategy has blind spots. Peer review of the strategy, using a structured checklist, catches errors before they compromise the review.

Peer review checks:

- are all key concepts represented?
- are major synonyms and variant spellings included?
- is the controlled vocabulary used correctly (explode, focus, major heading)?
- are operators and nesting syntactically correct?
- are there obvious omissions or over-restrictions?
- could another searcher reproduce the query from the documentation?

Tools exist for structured peer review of searches. Even informal peer review by a colleague meaningfully improves quality.

### Manage Search Versions And Updates

Searches are run, revised, re-run, and updated over the life of a project. Version control prevents confusion about which search produced which results.

Manage versions by:

- dating and labeling each search version;
- recording what changed between versions and why;
- keeping the results count for each version;
- storing query files with clear naming;
- planning the update search for the period before publication.

A reviewer who cannot tell which search is final cannot trust the evidence base.

### Run Updates Consistently

For living reviews, guidelines, and ongoing projects, the search must be updated for new literature. Updates must be consistent with the original to be comparable.

Update by:

- re-running the original strategy unchanged for the new date range;
- recording the date of the update search and results retrieved;
- screening new results against the same eligibility criteria;
- documenting any changes to the strategy and why.

Changing the strategy mid-project makes before-and-after comparisons invalid.

### Account For Known Coverage Biases

No database is complete. Reproducible searching requires acknowledging and mitigating coverage gaps so the evidence base is not silently distorted.

Account for:

- database overlap and unique coverage (run more than one for systematic work);
- under-representation of non-English, gray, and Global South literature;
- the limits of any single citation index for forward tracing;
- the need for trial registries, hand-searching, and author contact to find unpublished or ongoing work.

Documenting the gaps is part of an honest, reproducible search.

### Provide Enough Detail For Reproduction

The test of reproducibility is whether a competent searcher, given your documentation, could re-run the search and obtain comparable results. Write to that standard.

Include:

- the full strategy per database, not a summary;
- the database versions and dates;
- the filters and limits;
- the results counts;
- any deviations from the protocol and their rationale.

## Common Traps

### Running The Search Before Planning The Protocol

Ad hoc searching cannot be reconstructed. Plan and document the protocol first.

### Under-Reporting Methods

"We searched the literature" is not a method. Name sources, dates, limits, and results.

### Treating One Database As Sufficient For Systematic Work

No single database covers everything. Run multiple, plus gray literature and registries.

### Not Recording The Exact Query String

A paraphrased strategy is not reproducible. Capture the verbatim query and search history per database.

### Ignoring Database Version And Interface

Interfaces and versions change results. Record the platform and date, not just the database name.

### Changing The Strategy Mid-Project

Altering the search between runs breaks comparability. Lock the strategy and update it consistently.

### Skipping Peer Review Of The Strategy

Solo strategies have blind spots. Have a colleague or checklist review the search.

### No Version Control

Without labeled versions, no one knows which search is final. Date and label every version.

## Self-Check

- Did you define the protocol (question, eligibility, sources, strategy, limits) before running the search?
- Did you decompose the question into documented concepts, synonyms, and controlled vocabulary?
- Did you record the exact verbatim query string per database, with platform version and run date?
- Does the methods section report every source, date range, limits, and results counts transparently?
- Was the search strategy peer-reviewed against a structured checklist?
- Are search versions labeled, dated, and stored so the final version is unambiguous?
- For updates, did you re-run the original strategy unchanged for the new date range?
- Did you account for coverage biases and supplement with gray literature, registries, and hand-searching?
- Could a competent searcher reproduce this search and obtain comparable results from your documentation alone?
- Is the documentation detailed enough to survive peer review of the search methods?
