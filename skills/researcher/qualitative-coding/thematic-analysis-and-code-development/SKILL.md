# Thematic Analysis And Code Development

Thematic analysis is the most widely used qualitative method, and because it appears flexible and accessible, it is also the most frequently done badly. Agents often treat thematic analysis as reading the data, highlighting recurring ideas, and summarizing them, then present the summaries as findings. This produces a description of the data, not an analysis, and the description usually reflects the researcher's prior categories more than the participants' meaning. Thematic analysis is in fact a systematic method with decisions at every stage: whether codes are generated inductively from the data or deductively from theory, how codes are organized into themes, how themes are reviewed against the dataset, and how the analysis moves from description to interpretation. The judgment problem is to make these decisions deliberately, to develop a codebook that is grounded yet analytic, and to ensure that themes represent patterns in the data rather than the researcher's assumptions. A thematic analysis that skips these decisions produces findings that are untraceable, unreproducible, and often wrong.

Use this skill when conducting thematic analysis or developing a codebook. The goal is to prevent the agent from treating coding as casual highlighting, from confusing description with analysis, and from imposing themes that the data do not support. The agent has freedom in approach, but the coding decisions and the link from data to theme must be explicit and auditable.

## Core Rules

### Choose Between Inductive And Deductive Coding Deliberately

Whether codes arise from the data or from prior theory is a foundational decision that shapes the entire analysis.

Choose by:

- inductive (bottom-up) coding when the goal is to surface patterns grounded in participants' meaning;
- deductive (top-down) coding when the goal is to apply or test an existing framework;
- hybrid coding when both are used, with each portion labeled;
- consistency between the coding approach and the research question and claims.

Inductive coding presented as if it tests a theory, or deductive coding presented as if it is purely emergent, misrepresents what the analysis did and what it can claim.

### Develop A Codebook With Clear Definitions

A codebook is the instrument of thematic analysis. Without it, coding is private and unreproducible.

Develop by:

- a label and a clear definition for each code;
- inclusion criteria: what the code captures;
- exclusion criteria: what it does not capture;
- exemplar data excerpts for each code.

A code defined only by a label means whatever the coder thinks on a given day. Definitions and examples make coding explicit, consistent, and auditable.

### Code At The Right Grain Size

Codes that are too broad collapse distinct ideas; codes that are too narrow fragment the data into unusable fragments.

Calibrate by:

- starting close to the data with fine-grained initial codes;
- grouping and refining into more analytic codes as patterns emerge;
- avoiding codes so broad they capture everything and explain nothing;
- revising the codebook iteratively as understanding deepens.

The grain size should serve the analysis: fine enough to distinguish meaning, coarse enough to accumulate into patterns.

### Move From Codes To Themes Analytically

Themes are not just groups of related codes. They are analytic claims about meaningful patterns.

Move by:

- examining how codes cluster into coherent patterns;
- testing candidate themes against the full dataset;
- checking that themes capture something meaningful about the question;
- distinguishing central themes from minor ones.

A theme that is merely "all the codes about X" is a category, not a finding. A theme should offer an analytic claim about what the pattern means.

### Review Themes Against The Whole Dataset

Candidate themes must be checked against the entire dataset, not just the excerpts that inspired them.

Review by:

- whether the theme holds across the dataset or only in a few cases;
- whether the theme's boundaries are coherent;
- whether internal homogeneity and external heterogeneity are satisfied;
- whether some data are unaccounted for and what that means.

A theme built from vivid excerpts but untested against the full data may be an artifact of selective attention. Review against the whole dataset is essential.

### Ground Themes In Data With Evidence

Themes must be traceable to specific data. A theme asserted without evidence is the researcher's opinion.

Ground by:

- multiple supporting excerpts for each theme;
- excerpts that show the range and variation within the theme;
- explicit links from theme to the coded data;
- acknowledgment of disconfirming evidence.

A reader should be able to see, in the data, the basis for each theme. Themes presented without grounding cannot be assessed for credibility.

### Preserve Context In Coding And Quoting

Extracting excerpts from context can distort their meaning. Context must be preserved in analysis and reporting.

Preserve by:

- coding excerpts with enough surrounding context to retain meaning;
- noting the conditions and setting of each excerpt;
- avoiding quotes stripped of context in ways that change interpretation;
- representing participants' meaning rather than the researcher's gloss.

A quote removed from its narrative or setting may support a theme it actually complicates. Context is part of the meaning.

### Move From Description To Interpretation

Thematic analysis should not stop at describing what was said. It should interpret what the patterns mean.

Move by:

- describing the pattern clearly first;
- interpreting its significance for the question and theory;
- connecting themes to each other and to broader concepts;
- acknowledging the level of interpretation being offered.

Pure description is a starting point, not a contribution. The interpretive move is what makes the analysis analytic rather than a summary.

## Common Traps

### Coding As Casual Highlighting

Reading and summarizing without a codebook produces untraceable, unreproducible description, not analysis.

### Inductive And Deductive Confused

Presenting deductive coding as emergent, or inductive as theory-testing, misrepresents the analysis and its claims.

### Undefined Codes

Codes without definitions and examples mean whatever the coder thinks, undermining consistency and audit.

### Themes As Mere Code Groups

Grouping related codes and calling it a theme yields categories, not analytic findings.

### Selective Excerpt Bias

Building themes from vivid excerpts without testing against the full dataset produces artifacts of attention.

### Ungrounded Themes

Asserting themes without supporting data presents opinion as finding.

### Description Mistaken For Analysis

Stopping at what was said, without interpreting what it means, leaves the contribution undeveloped.

## Self-Check

- Is the choice between inductive, deductive, or hybrid coding deliberate and matched to the question?
- Does the codebook define each code with inclusion and exclusion criteria and exemplars?
- Is the coding grain size calibrated to distinguish meaning without fragmenting the data?
- Are themes developed as analytic claims about patterns, not just groups of related codes?
- Are candidate themes reviewed against the full dataset, not only the inspiring excerpts?
- Is each theme grounded in multiple data excerpts showing range and variation?
- Is context preserved in coding and quoting so meaning is not distorted?
- Does the analysis move from description to interpretation of what the patterns mean?
- Are disconfirming data acknowledged and accounted for?
- Can a reader trace each theme back to the coded data that supports it?
