# Reproducibility And Reproducible Analysis Practices

A statistical analysis that cannot be reproduced, that cannot be re-run by another researcher to produce the same results, is an analysis whose credibility rests entirely on trust in the original analysts. This is a fragile foundation. Reproducibility, the ability to re-run an analysis from data and code to results, is what allows the scientific community to verify findings, catch errors, build on prior work, and trust published results. Yet many published analyses are not reproducible: the data is unavailable, the code is undocumented or absent, the analysis decisions are under-specified, and the computational environment cannot be reconstructed. The reproducibility crisis in science is substantially a crisis of analyses that cannot be checked, because the materials needed to check them do not exist or were never shared. The judgment problem is conducting analyses reproducibly from the start, through version control, literate programming, documentation, and sharing, rather than treating reproducibility as a reporting afterthought that can be addressed if requested.

Use this skill when making analyses reproducible, using version control and literate programming, sharing data and code, documenting decisions, pre-registering, or diagnosing reproducibility failures. The goal is to prevent the agent from producing analyses that cannot be checked, from making undocumented decisions, from relying on proprietary or unavailable tools, and from treating reproducibility as optional.

## Core Rules

### Treat Reproducibility As A Design Principle, Not An Afterthought

Reproducibility is far easier to build in from the start than to reconstruct at the end. An analysis conducted with reproducibility in mind, with version control, documented code, and tracked decisions, is reproducible by construction. An analysis conducted informally and then made reproducible after the fact requires reconstructing decisions, environments, and data processing that were never recorded, often imperfectly.

Build in from the start by:

- using version control, such as Git, from the first analysis step, not added later;
- writing analysis as code, not interactive point-and-click, so it can be re-run;
- documenting data processing and analysis decisions as they are made;
- tracking the computational environment so it can be reconstructed;
- treating the analysis as a product to be shared, not a private process.

An analysis reconstructed for reproducibility after completion is a reconstruction, with the gaps and errors reconstruction entails.

### Use Version Control For All Analysis Code And Materials

Version control tracks every change to code and materials, providing a complete history of the analysis's evolution. This history is essential for reproducibility, because it shows what was done when, enables rollback to any state, and documents the analysis's development. Analysis without version control has no reliable history.

Use version control by:

- committing analysis code, scripts, and documentation to version control regularly;
- writing meaningful commit messages that explain what changed and why;
- using branches for exploratory analyses so the main analysis line stays clean;
- versioning data and results where feasible, or documenting data versions used;
- treating the version control history as part of the analysis record.

An analysis whose code exists only as a final file, with no history of how it developed, has lost the record of its evolution.

### Write Analysis As Code, Not Interactive Sessions

Interactive analysis, clicking through menus or typing commands in a session, produces results that cannot be re-run because the sequence of actions was not captured. Writing analysis as code, scripts or notebooks that can be executed end-to-end, captures the full analysis and makes it reproducible. Every transformation, model, and figure should be generated by code.

Write as code by:

- scripting the full analysis from data import through results, not just key steps;
- avoiding manual data editing or point-and-click transformations that are not captured;
- using literate programming tools, such as R Markdown or Jupyter, that combine code, narrative, and output;
- ensuring the script runs end-to-end from raw data to final results without intervention;
- avoiding hardcoded paths or manual steps that prevent re-running on another system.

An analysis that requires remembering which menu option was clicked is not reproducible; an analysis that runs from a script is.

### Document Analysis Decisions And Their Rationale

Reproducibility is not just re-running code; it is understanding why the code does what it does. Analysis decisions, why this transformation, this model, this subset, are often under-documented, leaving future readers able to re-run the analysis but not to understand it. Document decisions and their rationale.

Document by:

- writing narrative explanation alongside code, in literate programming or comments;
- explaining non-obvious transformations, exclusions, and modeling choices;
- distinguishing pre-specified from data-driven decisions;
- recording alternatives considered and rejected, and why;
- treating the documentation as part of the analysis, not optional commentary.

Code that can be re-run but not understood is mechanically reproducible but intellectually opaque, and opacity limits scientific value.

### Share Data Code And Materials

Reproducibility requires that others can access the materials needed to re-run the analysis. Data, code, and documentation that exist only on the analyst's computer might as well not exist for reproducibility purposes. Sharing, through repositories or supplementary materials, is what makes reproducibility real.

Share by:

- depositing analysis code and documentation in a permanent repository with a DOI;
- sharing data in a repository where licensing and ethics permit, with clear documentation;
- using open, accessible formats that do not require proprietary software;
- providing a clear README or guide for re-running the analysis;
- respecting confidentiality and licensing constraints while sharing as much as possible.

An analysis whose data and code are unavailable is reproducible in principle but not in practice, and practical reproducibility is what matters.

### Manage The Computational Environment

An analysis depends on its computational environment: the programming language version, packages, and system dependencies. An analysis that runs today may not run in a year if package versions have changed. Managing and documenting the environment is essential for long-term reproducibility.

Manage by:

- recording the versions of all software and packages used;
- using package management or environment capture tools to record dependencies;
- using containers, such as Docker, to capture the full environment where useful;
- avoiding reliance on bleeding-edge or proprietary tools that may not persist;
- documenting the environment so it can be reconstructed.

An analysis that cannot run because a package updated and changed its behavior is no longer reproducible, and this happens routinely without environment management.

### Pre-Register Analyses To Distinguish Planned From Exploratory

Pre-registration, recording the analysis plan before data examination, distinguishes planned confirmatory analyses from exploratory ones conducted after seeing the data. This distinction is essential for interpreting results, because exploratory analyses are hypothesis-generating and should not be treated as confirmatory. Pre-registration makes the distinction verifiable.

Pre-register by:

- registering the analysis plan, including outcomes, models, and corrections, before analysis;
- using registered reports or pre-registration platforms where appropriate;
- reporting deviations from the pre-registered plan and their reasons;
- distinguishing confirmatory results, from pre-specified analyses, from exploratory findings;
- recognizing that analyses conducted after seeing data require extra skepticism.

An analysis presented as confirmatory but actually exploratory, because it was designed after seeing the data, claims a rigor it does not have.

### Enable Computational Reproducibility Testing

Computational reproducibility, whether the analysis produces the same results when re-run, should be tested, not assumed. Analyses that have never been re-run on a clean environment may have hidden dependencies, hardcoded paths, or manual steps that prevent reproduction. Testing reveals these before publication.

Enable testing by:

- re-running the analysis on a clean environment before publication to confirm it works;
- having a colleague attempt to reproduce the analysis using only shared materials;
- using automated pipelines or makefiles that run the full analysis from raw data;
- fixing the issues that prevent clean reproduction;
- treating successful reproduction as a prerequisite for publication, not a nice-to-have.

An analysis that has never been reproduced by anyone other than the original analyst has untested reproducibility, and the first attempt often fails.

## Common Traps

### Reproducibility As Afterthought

Reconstructing reproducibility at the end is incomplete. Build it in from the start.

### No Version Control

Analysis without version control has no reliable history. Use version control throughout.

### Interactive Analysis Not Captured As Code

Point-and-click analysis cannot be re-run. Write the full analysis as code.

### Undocumented Decisions

Code that runs but is not understood is opaque. Document decisions and rationale.

### Unshared Data And Code

Materials on one computer do not enable reproducibility. Share through repositories.

### Unmanaged Computational Environment

Package and version changes break reproduction. Record and manage the environment.

### No Pre-Registration Distinguishing Planned From Exploratory

Exploratory analyses presented as confirmatory claim false rigor. Pre-register and distinguish.

### Untested Computational Reproducibility

Analyses never re-run may have hidden failures. Test reproduction before publication.

## Self-Check

- [ ] Reproducibility is built in from the start through version control, coded analysis, and tracked decisions.
- [ ] All analysis code and materials are under version control with meaningful history.
- [ ] The full analysis is written as executable code, not interactive sessions or manual steps.
- [ ] Analysis decisions and their rationale are documented in narrative alongside code.
- [ ] Data, code, and materials are shared in permanent repositories with clear documentation.
- [ ] The computational environment, including software and package versions, is recorded and managed.
- [ ] Analyses are pre-registered to distinguish confirmatory from exploratory analyses.
- [ ] Computational reproducibility has been tested by re-running on a clean environment.
- [ ] The analysis runs end-to-end from raw data to final results without intervention.
- [ ] No analysis relies on proprietary tools, hardcoded paths, or manual steps that prevent reproduction.
