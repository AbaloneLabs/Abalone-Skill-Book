# Qualitative Coding And Trustworthiness

Coding is where qualitative data become findings, and it is where most qualitative studies are won or lost. The work is not mechanical labeling. It is iterative interpretation: deciding what segments of data mean, how they relate, and what pattern they form, while staying answerable to the data. Agents often code once, build a theme list, and report it, treating coding as a sorting task. This collapses the interpretive depth that qualitative analysis requires. Equally common is the opposite failure: treating coding as purely subjective and therefore beyond quality control, which abandons rigor. The judgment problem is to code in a way that is systematic and auditable without pretending the process is objective in the positivist sense. Trustworthiness is built through transparency, grounding in data, attention to disconfirming evidence, and a documented trail of analytic decisions.

Use this skill when coding qualitative data, building themes, or establishing the credibility of findings. The goal is to prevent the agent from producing themes that float free of the data or that reflect the researcher's assumptions more than the participants' meaning.

## Core Rules

### Code Iteratively, Not In A Single Pass

Qualitative coding develops over cycles. First-cycle codes capture initial meaning; second-cycle codes organize and synthesize.

Code by:

- first-cycle coding: descriptive and structural codes close to the data;
- second-cycle coding: pattern, focused, or axial codes that organize;
- revisiting earlier data as later insights emerge;
- allowing the code structure to evolve, not fixing it in advance.

A single coding pass locks in early impressions and misses patterns visible only after sustained engagement.

### Build And Refine A Codebook Explicitly

A codebook documents what each code means and how to apply it. It is the backbone of consistency.

Build by:

- defining each code with a name, definition, and inclusion criteria;
- providing exemplar data for each code;
- revising definitions as the analysis deepens;
- recording when and why codes are split, merged, or retired.

Without a codebook, coding drifts and cannot be audited. With one, decisions become traceable and debatable.

### Ground Themes In Data, With Traceable Linkages

Every theme should be traceable to the data that support it. Findings that cannot be grounded are assertions, not analysis.

Ground by:

- linking each theme to the coded segments that instantiate it;
- retaining the participants' language where it carries meaning;
- showing the range of instances, not only the most vivid;
- noting how many participants and instances support each theme.

A theme illustrated by one dramatic quote but ungrounded in a broader pattern is an anecdote dressed as a finding.

### Move From Description To Interpretation Deliberately

Description summarizes what the data say; interpretation explains what they mean. Both are needed; conflating them weakens analysis.

Move by:

- descriptive themes that organize the surface content;
- interpretive themes that explain patterns and relationships;
- explicit links between the descriptive and interpretive levels;
- avoiding the leap to interpretation without descriptive grounding.

Reporting only description produces a data summary; reporting only interpretation without grounding produces unsupported claims.

### Attend To Negative And Disconfirming Cases

Cases that do not fit the emerging pattern are not noise to be discarded. They test and refine the analysis.

Attend by:

- searching actively for cases that contradict the pattern;
- revising the theme to account for variation;
- reporting the boundaries of where the theme holds and does not;
- treating disconfirming evidence as a source of analytic insight.

A theme that survives an active search for disconfirmation is far more credible than one never tested.

### Use Multiple Coders To Surface Interpretive Judgment

Inter-coder agreement is not about objectivity; it is about making interpretive decisions visible and debatable.

Use by:

- coding independently on a subset of data;
- discussing disagreements rather than averaging them;
- revising the codebook to resolve recurring ambiguities;
- reporting agreement as evidence of process, not proof of truth.

High agreement achieved by trivial codes is meaningless; meaningful disagreement, discussed and resolved, strengthens the analysis.

### Maintain An Audit Trail Of Analytic Decisions

Trustworthiness depends on the ability of others to follow how the researcher moved from data to findings.

Maintain by:

- memos documenting analytic decisions and their rationale;
- a record of code evolution and theme development;
- decision rules for inclusion and exclusion;
- the chain of evidence from raw data to final claims.

An audit trail does not make interpretation objective, but it makes it inspectable, which is what trustworthiness requires.

### Use Reflexivity To Surface Researcher Influence

The researcher is the instrument in qualitative work. Their assumptions shape what is seen and coded.

Use by:

- reflexive memos on how the researcher's position affects coding;
- attention to how expectations may steer theme development;
- distinguishing participant meaning from researcher inference;
- acknowledging the partial, situated nature of the findings.

Findings presented as if the researcher had no position hide the interpretive work and overstate neutrality.

### Match The Level Of Claim To The Strength Of The Evidence

Qualitative claims must be proportionate to the data. Overclaiming is the most common threat to credibility.

Match by:

- claims about these participants in this context;
- theoretical or analytical generalization where supported;
- avoiding prevalence claims from small samples;
- acknowledging what the analysis cannot establish.

A theme from 12 interviews is insight into meaning and pattern, not a statement about how common a view is in a population.

## Common Traps

### Single-Pass Coding

Coding once locks in early impressions and misses patterns that emerge only through iterative engagement.

### No Codebook

Without documented code definitions, coding drifts and cannot be audited or defended.

### Themes Floating Free Of Data

Findings illustrated by vivid quotes but not grounded in a broader pattern are anecdotes, not analysis.

### Leaping To Interpretation Without Description

Jumping to meaning without first establishing what the data say produces unsupported interpretive claims.

### Ignoring Disconfirming Cases

Discarding cases that do not fit produces tidy but untested themes that collapse under scrutiny.

### Inter-Coder Agreement As Proof Of Truth

High agreement on trivial codes is meaningless; the value is in surfacing and resolving interpretive disagreement.

### Overclaiming From Small Samples

Extending qualitative findings to populations or prevalence misuses the approach and overstates the contribution.

## Self-Check

- Is coding conducted iteratively, with first- and second-cycle coding and revisiting of earlier data?
- Is there an explicit, evolving codebook with definitions, inclusion criteria, and exemplars?
- Are themes grounded in data, with traceable linkages to coded segments and participant language?
- Does the analysis move deliberately from description to interpretation, with both levels present?
- Are negative and disconfirming cases actively sought and used to refine themes?
- Are multiple coders used to surface interpretive judgment, with disagreements discussed?
- Is an audit trail maintained, documenting decisions, code evolution, and the chain of evidence?
- Is reflexivity used to surface how the researcher's position shapes coding and themes?
- Are claims matched to the strength of the evidence, avoiding prevalence claims from small samples?
- Are the boundaries of the findings, including what they cannot establish, acknowledged?
