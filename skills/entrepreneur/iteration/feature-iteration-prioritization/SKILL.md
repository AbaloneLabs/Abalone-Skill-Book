# Feature Iteration Prioritization

A startup always has more feature ideas than it can build, and choosing poorly is costly. Every feature built consumes scarce engineering time, adds complexity, and can dilute the core value. Good feature prioritization is not about satisfying every request; it is about choosing the few changes that most advance learning, retention, or strategy, and deferring or declining the rest. The discipline is to rank features by the evidence and impact they produce, not by opinion, urgency, or the volume of requests, and to revisit the ranking as learning changes what matters.

Use this skill before answering broad questions such as "what should I build next", "how do I prioritize features", "which feature should I ship first", or "how should an entrepreneur decide what to build". The goal is to prevent the agent from prioritizing by opinion or request volume and from letting the roadmap become a list of demands rather than a learning plan.

## Core Rules

### Prioritize Against The Current Goal

The right features depend on what the venture is trying to achieve right now: validating demand, improving retention, enabling a growth mechanism, or preparing to scale. Rank features by how much they advance the current goal. A feature valuable for a different goal is the wrong priority now, however attractive.

### Rank By Expected Learning And Impact

For each candidate feature, estimate the learning or impact it would produce: how much it would move the key metric, how much it would reduce uncertainty about an assumption, how many users it would help. Rank by expected value, not by effort alone or by who asked. Features with high expected impact and low effort go first.

### Separate Hypothesis-Driven From Request-Driven Work

Hypothesis-driven features test an assumption and produce learning; request-driven features satisfy a specific user's ask. Both can be valid, but the roadmap should be weighted toward hypothesis-driven work that advances the venture, with request-driven work filling genuine gaps. A roadmap that is only requests becomes a help desk, not a product strategy.

### Account For Effort And Opportunity Cost

A feature's cost is not only the engineering effort; it is the opportunity cost of what else that effort could build. A large feature that blocks several smaller, higher-impact ones may be the wrong choice even if it is valuable in isolation. Compare features against each other, not only against zero.

### Beware The Loudest Customer

A vocal customer or a few large accounts can distort prioritization toward their needs, which may not represent the market. Weight requests by whether they reflect a broad pattern or a single edge case. Building for the loudest voice produces a product optimized for a few users, not the segment.

### Prefer High-Learning Features Before Fit

Before product-market fit, prioritize features that produce learning about the core assumption over features that add polish or breadth. Learning features reduce uncertainty and move the venture toward fit; polish features optimize a product whose shape may still change. Match the feature type to the stage.

### Defer And Decline Deliberately

Saying yes to every request bloats the product and dilutes focus. Build the discipline to defer features to later and to decline features that do not serve the current goal, with clear reasoning. A roadmap that cannot decline is not prioritized; it is a queue.

### Re-Prioritize As Learning Changes What Matters

Prioritization is not a one-time sort. As cycles produce learning, what matters changes: an assumption is confirmed or refuted, a metric moves, a segment emerges. Re-prioritize the roadmap as learning accumulates, rather than executing a stale plan.

### Watch For Complexity Cost

Each feature adds complexity that makes the product harder to maintain, support, and improve. Account for the complexity cost when prioritizing: a feature that adds ongoing burden must justify it with sufficient impact. Some features are worth more declined than built.

### Acknowledge Personal Risk And Professional Advice

Feature decisions shape the product and carry stakes for venture direction and personal risk. Where features involve data, payments, or regulatory exposure, professional advice may be needed. This skill does not replace that counsel.

## Common Traps

### Prioritizing By Opinion Or Volume

Features chosen by who asks loudest advance the wrong things.

### Ignoring Opportunity Cost

A valuable feature in isolation can be the wrong choice if it blocks higher-impact work.

### Building For The Loudest Customer

Optimizing for a few vocal users misses the segment.

### Polish Before Fit

Adding finish to a product whose shape may still change wastes effort.

### Roadmap As A Queue

A backlog that cannot decline is not prioritized and bloats the product.

### Ignoring Complexity Cost

Each feature adds maintenance and support burden that must be justified.

## Self-Check

- [ ] Features are prioritized against the venture's current goal.
- [ ] Ranking considers expected learning and impact, not only effort or requests.
- [ ] Hypothesis-driven and request-driven work are distinguished and weighted appropriately.
- [ ] Effort and opportunity cost are accounted for in comparisons.
- [ ] The loudest customer's requests are weighted against broad patterns, not treated as decisive.
- [ ] Before fit, learning features are prioritized over polish.
- [ ] Features are deferred and declined deliberately, with reasoning.
- [ ] The roadmap is re-prioritized as learning changes what matters.
- [ ] Complexity cost is accounted for in prioritization.
- [ ] Data, payment, and regulatory exposures are flagged for professional advice.
