# Build-Measure-Learn Cycles

The build-measure-learn cycle is the engine of lean iteration: build the smallest thing that tests an assumption, measure the evidence it produces, learn what the evidence means, and let the learning drive the next build. The power is in the speed and honesty of the loop. Fast loops compress learning into the runway the venture has. Honest loops, where evidence is interpreted without rationalization, produce real learning. When the loop is slow, the venture learns too late; when it is dishonest, the venture learns nothing and mistakes activity for progress.

Use this skill before answering broad questions such as "how do I iterate on my product", "what is the build-measure-learn loop", "how do I run learning cycles", or "how should an entrepreneur structure product iteration". The goal is to prevent the agent from collapsing the loop into building and from letting measurement become vanity tracking rather than assumption testing.

## Core Rules

### Start The Cycle From An Assumption, Not A Feature

Each cycle should begin with a specific assumption to test, usually about the customer, the value, or the growth mechanism. Starting from a feature, "let's build X", without the assumption it tests, produces activity without learning. State the assumption first, then design the smallest build that tests it.

### Design The Build To Minimize Cycle Time

The build should be the smallest thing that produces credible evidence about the assumption. Smaller builds mean faster cycles and more learning within the runway. Resist building more than the test requires; every additional element lengthens the cycle without adding learning.

### Measure Against The Assumption, Not Against Activity

The measurement step should capture evidence about whether the assumption holds, not whether activity occurred. Define the metric that would confirm or refute the assumption before building, and measure it directly. Metrics disconnected from the assumption produce data without insight.

### Learn Through A Written, Committed Decision

The learning step is not a feeling; it is a written decision about what the evidence means and what changes as a result. If the evidence supports the assumption, proceed; if it refutes it, change the assumption, the product, or the direction. A cycle that ends without a committed decision has not completed the loop.

### Compress Cycle Time Deliberately

The value of the loop scales with speed. Identify what slows the cycle, long builds, slow measurement, indecision, and attack those bottlenecks. The goal is the maximum number of informed cycles the runway allows, because each cycle reduces uncertainty.

### Prevent Rationalization Of Disappointing Results

When evidence disappoints, the temptation is to rationalize: the sample was too small, the timing was wrong, the feature needs polish. Build discipline against rationalization by defining the evidence standard before the cycle and committing to act on it. A loop that explains away every failure learns nothing.

### Run Cycles In Parallel Where Possible

Not every cycle must be sequential. Some assumptions can be tested in parallel, compressing total learning time. Identify independent assumptions and run their cycles concurrently, while sequencing dependent ones. Parallelism multiplies learning within the same runway.

### Distinguish Validating From Exploring Cycles

Some cycles validate a specific assumption with a defined metric; others explore an unknown space to discover what matters. Both are legitimate, but they differ in design. Validating cycles need tight metrics and decisions; exploring cycles need broad observation and hypothesis generation. Know which kind of cycle is running.

### Feed Learning Back Into Strategy

The loop should inform not only the product but the strategy: who the customer is, what the value proposition is, what the model is. Cycles that only refine features miss strategic insights the evidence contains. Let learning reshape the venture's direction, not just its roadmap.

### Know When To Stop Cycling On An Assumption

Some assumptions, after enough cycles, are simply false. Continued cycling becomes denial. Define a stopping rule: how many cycles, or what pattern of results, would indicate the assumption is not holding. Be willing to pivot or stop rather than cycle indefinitely.

### Acknowledge Personal Risk And Professional Advice

Iteration cycles inform decisions with stakes for venture direction and personal risk. Where cycles involve customer data or claims, professional advice may be needed. This skill does not replace that counsel.

## Common Traps

### Starting From A Feature, Not An Assumption

Building without a stated assumption produces activity without learning.

### Measuring Activity Instead Of The Assumption

Vanity metrics produce data without insight.

### No Committed Decision

A cycle that ends without a decision has not completed the loop.

### Rationalizing Disappointing Results

Explaining away failure prevents learning.

### Slow Cycles Burning Runway

Long builds and slow measurement exhaust the runway before learning accumulates.

### Cycling Past A False Assumption

Continuing to refine an assumption cycles have shown to be false.

## Self-Check

- [ ] Each cycle begins from a specific assumption, not a feature idea.
- [ ] The build is the smallest credible test of that assumption.
- [ ] Measurement captures evidence about the assumption, not mere activity.
- [ ] The cycle ends with a written, committed decision.
- [ ] Cycle time is compressed by attacking bottlenecks.
- [ ] Rationalization of disappointing results is guarded against with pre-defined evidence standards.
- [ ] Independent assumptions are tested in parallel where possible.
- [ ] Validating and exploring cycles are distinguished and designed appropriately.
- [ ] Learning feeds back into strategy, not only into the feature roadmap.
- [ ] A stopping rule for cycling on an assumption is defined.
- [ ] Data and claims exposures are flagged for professional advice.
