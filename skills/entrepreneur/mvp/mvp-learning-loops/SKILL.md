# MVP Learning Loops

An MVP is not a one-shot experiment; it is the start of a learning loop. Each version generates evidence, the evidence informs a decision, the decision changes the product or the assumption, and the next version tests the revised belief. The value of an MVP comes from how fast and how honestly this loop turns. Slow loops delay learning; dishonest loops, where evidence is rationalized or ignored, produce no learning at all. Designing the loop, the metrics, the decision points, and the cadence, is what turns an MVP from a single test into a sustained engine of discovery.

Use this skill before answering broad questions such as "how do I iterate on my MVP", "what should I measure after launch", "how do I learn from my MVP", or "how should an entrepreneur run MVP learning cycles". The goal is to prevent the agent from treating MVP launch as the end and from letting iteration become feature accumulation without evidence or decisions.

## Core Rules

### Define The Loop Before The MVP Launches

The learning loop should be designed before the MVP ships: what assumption each cycle tests, what metric indicates success or failure, what decision follows each result, and how long each cycle runs. Defining the loop in advance prevents post-launch drift, where the team reacts to whatever is loudest rather than to the planned learning goal.

### Measure Against The Assumption, Not Against Activity

Each cycle should measure whether the assumption being tested is supported, not whether activity is happening. Sign-ups, clicks, and downloads are activity; activation, retention, willingness to pay, and referral are evidence about the assumption. Choose metrics that speak to the belief being tested.

### Set Decision Points And Commit To Them

For each cycle, define the decision that follows: if the metric crosses a threshold, proceed; if it falls short, change the product, change the assumption, or stop. Committing to the decision in advance prevents the team from moving the goalposts when results are disappointing. A loop without committed decisions is not a loop; it is observation.

### Run Cycles Fast Enough To Learn

The speed of the loop determines how much the venture can learn within its runway. Long cycles, slow builds, slow measurement, produce little learning before money runs out. Compress cycles by building less, measuring sooner, and deciding faster. The goal is the maximum number of informed cycles the runway allows.

### Distinguish Learning Iteration From Feature Accumulation

Iteration driven by evidence changes the product to test a revised assumption. Feature accumulation adds capabilities because they seem useful or because customers ask, without a learning goal. The first advances understanding; the second bloats the product and burns time. Keep iteration tied to specific assumptions.

### Close The Loop With A Written Decision

Each cycle should end with a written decision: what was learned, what it means, and what changes as a result. Writing forces honesty and creates a record that prevents the team from repeating tests or forgetting conclusions. A loop without written decisions loses its accumulated learning.

### Watch For False Positives And False Negatives

A positive result may reflect a confound, a particularly friendly cohort, a novelty effect; a negative result may reflect a build problem, a targeting error, or insufficient time. Before acting on a cycle's result, consider whether confounds could explain it. A single cycle's evidence is weak; patterns across cycles are stronger.

### Know When To Stop Iterating On An Assumption

Some assumptions, after enough cycles, are simply false, and continued iteration becomes denial. Define in advance how many cycles or what pattern of results would indicate the assumption is not holding, and be willing to pivot or stop rather than iterate indefinitely on a failing belief.

### Feed Learning Back Into Strategy, Not Only Product

MVP learning should inform not just the product but the strategy: who the customer is, what the value proposition is, what the model is. Loops that only refine features miss the strategic insights the evidence may contain. Let learning reshape the venture's direction, not just its feature list.

### Acknowledge Personal Risk And Professional Advice

Learning loops inform decisions with stakes for venture direction and personal risk. Where cycles involve customer data or claims, professional advice may be needed. This skill does not replace that counsel.

## Common Traps

### Launch Treated As The End

Shipping the MVP without a planned loop produces observation, not learning.

### Measuring Activity, Not Assumptions

Tracking clicks and sign-ups that do not speak to the belief being tested.

### No Committed Decisions

A loop without decision points lets the team move goalposts and avoid hard conclusions.

### Cycles Too Slow

Long builds and slow measurement exhaust runway before learning accumulates.

### Feature Accumulation Mistaken For Iteration

Adding features without a learning goal bloats the product and tests nothing new.

### Iterating Past A False Assumption

Continuing to refine an assumption that enough cycles have shown to be false.

## Self-Check

- [ ] The learning loop, metrics, decisions, and cadence, is defined before the MVP launches.
- [ ] Each cycle measures evidence about the assumption, not mere activity.
- [ ] Decision points are set and committed in advance for each cycle.
- [ ] Cycles run fast enough to learn within the runway.
- [ ] Iteration is driven by evidence about assumptions, not feature accumulation.
- [ ] Each cycle ends with a written decision that is recorded.
- [ ] Possible confounds, false positives and false negatives, are considered before acting.
- [ ] A stopping rule for iterating on an assumption is defined.
- [ ] Learning feeds back into strategy, not only into the product feature list.
- [ ] Data and claims exposures during cycles are flagged for professional advice.
