# MVP Build Decisions

Once the MVP's scope is defined, the founder faces build decisions: what to construct, what to assemble from existing tools, how much quality to invest, and how to balance speed against robustness. These decisions are easy to get wrong in both directions. Over-building produces a polished, scalable product that tests nothing and burns runway; under-building produces something so fragile that its failure reflects the build, not the assumption. Good build decisions serve the learning goal: they produce credible evidence fast, using the cheapest reliable means, and they defer polish and scale until evidence justifies them.

Use this skill before answering broad questions such as "how should I build my MVP", "should I code it or use no-code", "how much should I invest in MVP quality", or "how should an entrepreneur make MVP build choices". The goal is to prevent the agent from optimizing the build for the wrong qualities, such as scalability or polish, before the assumption is tested.

## Core Rules

### Optimize The Build For Learning, Not For Polish

The MVP build should be optimized to produce evidence about the assumption, not to look impressive or feel complete. Polish, branding, and finish that do not change the evidence are wasted effort at this stage. Build the minimum quality that lets the evidence be credible; defer the rest.

### Assemble Before You Build

Before writing custom software, check whether existing tools, no-code platforms, templates, or manual processes can test the assumption. Assembly is faster and cheaper than building, and for many assumptions it produces equally credible evidence. Reserve custom building for the parts no existing tool can test.

### Match Build Quality To The Evidence Required

The right quality is the lowest that produces credible evidence. A concierge MVP delivered manually can be rough; an MVP that takes payments must be reliable enough to transact safely. Match quality to what the evidence demands, not to an abstract standard of good engineering.

### Avoid Premature Scalability

Building for scale before demand is proven is a common waste. An MVP that serves dozens of users does not need the architecture of one serving millions. Build for the scale the test requires, and plan to rebuild for scale only when evidence justifies it. Premature scalability delays learning and burns runway.

### Decide Code Versus No-Code By The Learning Goal

Code is justified when the learning requires behavior no existing tool provides. No-code and assembled tools are justified when they can produce the evidence faster. Choose by the learning goal, not by the founder's comfort with a technology or a desire to build "real" software.

### Time-Box The Build

Builds expand to fill available time. Set a fixed time box for the MVP build and ship what is ready. A time box forces decisions about what truly matters for learning and prevents endless refinement that delays evidence.

### Plan For Throwaway And Rebuild

Accept that much of the MVP will be thrown away or rebuilt once learning changes the design. Building the MVP as if it were the foundation of the final product leads to over-engineering. Build to learn, then rebuild to scale and last.

### Ensure The Build Does Not Confound The Test

If the MVP is too slow, too buggy, or too ugly, its failure may reflect the build, not the assumption. Ensure the build quality is high enough that a negative result can be attributed to the assumption being false, not to the product being unusable. Confounding undermines the evidence.

### Involve Real Users In The Build Loop

Build in small increments and expose them to real users early, so the build incorporates feedback before it hardens. A build loop isolated from users produces a product that meets the founder's assumptions, not the users' needs.

### Acknowledge Personal Risk And Professional Advice

Build decisions involving payments, data, or integrations may carry security, privacy, and regulatory obligations that require professional advice. This skill does not replace that counsel.

## Common Traps

### Over-Building For Polish

A polished MVP that tests nothing burns runway and delays evidence.

### Premature Scalability

Architecture for millions when the test needs dozens wastes effort.

### Building When Assembly Would Suffice

Custom code where no-code or existing tools would test the assumption faster.

### Confounding Build Quality With Assumption Failure

A negative result blamed on the assumption when the MVP was simply unusable.

### No Time Box

Builds without a fixed end expand and delay learning indefinitely.

### Treating The MVP As The Foundation

Over-engineering the MVP as if it must last leads to rigidity and waste.

## Self-Check

- [ ] The build is optimized for producing evidence, not for polish or completeness.
- [ ] Existing tools and assembly are considered before custom building.
- [ ] Build quality matches the minimum needed for credible evidence.
- [ ] Scalability is deferred until demand is proven.
- [ ] Code versus no-code is chosen by the learning goal, not comfort or pride.
- [ ] The build is time-boxed to force shipping.
- [ ] The plan accepts throwaway and rebuild once learning changes the design.
- [ ] Build quality is high enough that failure reflects the assumption, not unusability.
- [ ] Real users are involved in the build loop before design hardens.
- [ ] Payment, data, and integration obligations are flagged for professional advice.
