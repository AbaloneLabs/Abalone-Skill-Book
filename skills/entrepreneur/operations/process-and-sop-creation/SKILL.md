# Process And SOP Creation

Processes and standard operating procedures are how recurring work gets done consistently, especially as it moves beyond the founder. Documenting a process turns tacit knowledge, held in one person's head, into transferable knowledge that others can execute. The value is consistency, delegation, and quality at scale. The risks are two-sided: documenting too little leaves the venture dependent on individuals and prone to breakdown; documenting too much, or too early, creates rigid bureaucracy that slows a team that needs flexibility. The discipline is to formalize the work that is repeated, important, and transferable, and to keep documentation living rather than stale.

Use this skill before answering broad questions such as "how do I create an SOP", "what processes should I document", "how do I make work repeatable", or "how should an entrepreneur formalize their processes". The goal is to prevent the agent from over-documenting prematurely and from creating procedures that do not match how work actually happens.

## Core Rules

### Document What Is Repeated, Important, And Transferable

Not all work needs documentation. Prioritize documenting work that is repeated often, important to quality or compliance, and transferable to someone else. One-off or creative work rarely benefits from rigid process; recurring operational work does. Focus documentation where it unlocks consistency and delegation.

### Document Real Workflows, Not Idealized Ones

A procedure that describes how work should happen in theory is ignored; a procedure that describes how work actually happens is used. Document the real workflow, including the workarounds and judgment currently applied, then improve it. Idealized documentation that no one follows is worse than no documentation, because it creates false confidence.

### Create Documentation To Enable Delegation

The primary purpose of formalizing a process is to let someone else do the work reliably. Write documentation with the next executor in mind: what they need to know, what decisions they can make, what to do when something deviates. If the documentation cannot guide a new person, it has not achieved its purpose.

### Capture The Decisions And Exceptions, Not Only Steps

Steps are easy to document; the judgment is harder. Capture the decisions within the process, when to take which path, and how to handle exceptions and edge cases. A procedure that lists only steps breaks the moment reality deviates; one that captures decision rules adapts. The judgment is often the most valuable part of the documentation.

### Keep Documentation Living

Processes change as the venture, tools, and team evolve. Documentation that is not updated becomes stale and misleading, describing work that no longer matches reality. Assign ownership for keeping key documentation current, and review it regularly. Stale documentation erodes trust in all documentation.

### Avoid Premature Formalization

Formalizing a process before it has stabilized locks in a workflow that may still be changing. Let a process settle into a repeatable pattern through practice before documenting it, so the documentation captures what works rather than a transient experiment. Premature formalization creates rework and rigidity.

### Balance Consistency With Judgment

Processes exist to ensure consistency, but not all variation is bad. Over-rigid procedures remove the judgment that experienced team members apply to edge cases. Build procedures that ensure consistency on the critical steps while leaving room for judgment where variation is appropriate.

### Make Documentation Discoverable And Usable

Documentation that cannot be found or understood is useless. Make procedures discoverable in a shared, organized location, written clearly, with the right level of detail for the executor. Usability, searchability, and clarity determine whether documentation is actually used.

### Use Documentation To Improve, Not Only To Record

Documenting a process reveals inefficiencies, gaps, and risks that improvisation hides. Use documentation as an opportunity to improve the process, not only to record it. The act of formalizing often surfaces improvements that make the work better.

### Acknowledge Personal Risk And Professional Advice

Processes may involve compliance, safety, employment, and contractual obligations with legal implications that require professional advice. This skill does not replace that counsel.

## Common Traps

### Over-Documenting Prematurely

Formalizing processes before they stabilize locks in transient workflows.

### Idealized Documentation

Procedures describing theory, not reality, are ignored and erode trust.

### Steps Without Judgment

Procedures that list only steps break when reality deviates.

### Stale Documentation

Unupdated procedures mislead and reduce trust in all documentation.

### Documentation That Cannot Be Found

Undiscoverable or unclear documentation is effectively useless.

### Rigidity Removing Judgment

Over-rigid procedures remove the judgment needed for edge cases.

## Self-Check

- [ ] Documentation prioritizes work that is repeated, important, and transferable.
- [ ] Procedures describe real workflows, including workarounds, not idealized theory.
- [ ] Documentation is written to enable delegation to a new executor.
- [ ] Decisions, exceptions, and judgment are captured, not only steps.
- [ ] Documentation is kept living, with ownership and regular review.
- [ ] Processes are allowed to stabilize before formalization.
- [ ] Procedures balance consistency on critical steps with room for judgment.
- [ ] Documentation is discoverable, organized, and clearly written.
- [ ] Documentation is used as an opportunity to improve the process.
- [ ] Compliance, safety, employment, and contractual implications are flagged for professional legal advice.
