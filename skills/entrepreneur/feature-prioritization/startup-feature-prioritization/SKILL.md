# Startup Feature Prioritization

A startup roadmap is a statement of strategy expressed as a sequence of choices about what to build. Because engineering capacity is always smaller than the list of ideas, every feature chosen is another deferred, and the quality of those choices determines whether the product moves toward fit and differentiation or drifts into bloat. Feature prioritization is the discipline of ranking candidates by their expected contribution to the venture's current goal, using evidence rather than opinion, and revising the ranking as learning changes what matters.

Use this skill before answering broad questions such as "how do I prioritize my roadmap", "what framework should I use for features", "how do I decide what to build", or "how should an entrepreneur prioritize product work". The goal is to prevent the agent from recommending generic frameworks applied without context and from letting prioritization become reaction to the loudest demand.

## Core Rules

### Anchor Prioritization To The Current Strategic Goal

The roadmap must serve the venture's current goal: validating demand, reaching retention, enabling growth, or preparing to scale. Rank features by how much they advance that goal. A feature valuable for a different goal is a distraction now. Re-anchor prioritization whenever the strategic goal shifts.

### Score Features On Expected Value, Not On Opinion

Use a scoring approach that captures expected value: the impact a feature would have on the key metric or assumption, weighted by confidence and divided by effort. The point is not the precise score but the discipline of comparing features on consistent dimensions rather than by gut. Features with high expected value and low effort rise; vague high-effort ideas sink.

### Account For Confidence And Evidence

A feature's expected impact is only as credible as the evidence behind it. Weight scores by confidence: features backed by user behavior, retention data, or test results deserve higher confidence than features backed by opinion or request. Distinguish what is known from what is guessed when ranking.

### Treat Effort As Opportunity Cost

Effort is not only the cost of a feature; it is the value of the other features that effort could have built. A feature that consumes a quarter of capacity blocks several alternatives. Compare features against each other, not against zero, and ask what each choice forecloses.

### Separate The Core From The Edges

Prioritize features that strengthen the core value loop over features that extend breadth. Core features compound, improving retention and differentiation; edge features add surface area without deepening value. Before fit, the core is almost all that matters; after fit, edges can extend, but the core still leads.

### Decline And Defer Deliberately

A roadmap that cannot say no is not prioritized. Build the discipline to decline features that do not serve the current goal and to defer features that do but not now. Clear reasoning for each decline keeps the roadmap strategic and prevents backlog drift into the product.

### Re-Prioritize On A Regular Cadence

Prioritization decays as learning accumulates and conditions change. Re-rank the roadmap on a regular cadence, incorporating new evidence, shifting goals, and changing constraints. A roadmap executed without re-prioritization reflects a past understanding, not the current one.

### Beware HiPPO And Loud Voices

The highest-paid person's opinion or the loudest customer's request can override evidence-based ranking. Guard against this by making the scoring and reasoning explicit, so that prioritization is defensible and not reducible to authority or volume. Explicit criteria discipline opinion.

### Plan For Learning Features Alongside Delivery Features

Some features exist to learn, not to deliver; they test an assumption and may be discarded. Allocate capacity to learning features, especially before fit, because they reduce uncertainty that delivery features cannot. A roadmap of only delivery features assumes the strategy is already correct.

### Acknowledge Personal Risk And Professional Advice

Roadmap decisions shape the product and carry stakes for venture direction and personal risk. Where features involve data, payments, or regulatory exposure, professional advice may be needed. This skill does not replace that counsel.

## Common Traps

### Generic Framework Without Context

A scoring template applied without strategic anchoring produces ranked busywork.

### Prioritizing By Opinion Or Authority

HiPPO and loud voices override evidence and distort the roadmap.

### Ignoring Opportunity Cost

A feature valuable in isolation blocks higher-impact alternatives.

### Breadth Over Core

Extending surface area before deepening the core dilutes value.

### Roadmap That Cannot Decline

A backlog that becomes the product produces bloat and lost focus.

### Static Roadmap

Executing a plan without re-prioritization reflects past, not current, understanding.

## Self-Check

- [ ] Prioritization is anchored to the venture's current strategic goal.
- [ ] Features are scored on expected value, confidence, and effort, not opinion.
- [ ] Confidence reflects the evidence behind each feature's expected impact.
- [ ] Effort is treated as opportunity cost, comparing features against each other.
- [ ] Core value loop features are prioritized over breadth extensions.
- [ ] Features are declined and deferred deliberately with clear reasoning.
- [ ] The roadmap is re-prioritized on a regular cadence.
- [ ] HiPPO and loud voices are guarded against with explicit criteria.
- [ ] Capacity is allocated to learning features alongside delivery features.
- [ ] Data, payment, and regulatory exposures are flagged for professional advice.
