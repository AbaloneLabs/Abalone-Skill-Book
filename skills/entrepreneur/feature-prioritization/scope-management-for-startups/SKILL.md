# Scope Management For Startups

Scope is what the venture commits to build, deliver, or support. Left unmanaged, scope expands to fill available time and beyond: features accrete, commitments multiply, and the product grows complexity faster than capacity. Scope management is the discipline of keeping what the venture commits to aligned with its capacity, runway, and evidence. It is not about thinking small; it is about choosing deliberately so that the venture actually finishes what matters, rather than starting many things and completing few.

Use this skill before answering broad questions such as "how do I manage scope", "how do I stop feature creep", "how do I keep my product focused", or "how should an entrepreneur control what they commit to". The goal is to prevent the agent from equating ambition with large scope and from letting scope expand without a forcing function for commitment.

## Core Rules

### Treat Scope As A Commitment, Not A Wishlist

Scope is what the venture commits to deliver, with the capacity and time to actually finish it. A list of desired features is a wishlist, not a scope. Define scope by what will be done, with a clear boundary against what will not, and resource it realistically. Confusing wishlist with scope produces chronic overruns.

### Match Scope To Capacity And Runway

Scope must fit the capacity available within the runway. A scope that requires more engineering, time, or money than the venture has is not a plan; it is a setup for failure. Match scope to realistic capacity, including the overhead of support, maintenance, and iteration, not only new building.

### Define Explicit Out-Of-Scope Boundaries

For each release or phase, define not only what is in scope but what is explicitly out of scope. Explicit out-of-scope boundaries prevent drift, because every new request can be measured against them. Without explicit boundaries, scope creeps through accretion of "small" additions.

### Use A Forcing Function For Commitment

Scope expands without pressure to commit. Use a forcing function, a time box, a milestone, a launch date, a funding event, to force decisions about what truly belongs in the scope. The forcing function turns a wishlist into a committed scope by requiring tradeoffs.

### Resist Scope Creep From Multiple Sources

Scope creeps from many sources: customer requests, investor suggestions, competitive anxiety, founder ideas, and engineering preferences. Each addition seems small in isolation, but together they bloat the scope. Evaluate each addition against the current goal and the explicit boundaries, and decline those that do not serve the goal.

### Distinguish Core From Incremental Scope

Core scope is what the release must have to achieve its goal; incremental scope is what would be nice if capacity allows. Build the core first, and add incremental scope only if the core finishes with margin. This sequencing protects the essential from being delayed by the optional.

### Account For Hidden Scope

Hidden scope, the support, documentation, testing, migration, communication, and maintenance that each feature requires, is rarely counted but often larger than the feature itself. Account for hidden scope when committing, or the release will overrun on work that was never visible.

### Re-Scope When Conditions Change

Scope set under one set of assumptions becomes wrong when conditions change: capacity drops, evidence shifts, priorities move. Re-scope when conditions change, rather than executing a scope that no longer fits reality. Clinging to an obsolete scope produces failure by overrun.

### Communicate Scope And Changes Clearly

Scope and changes to it should be communicated clearly to everyone affected: the team, stakeholders, and where relevant, customers. Surprising people with scope cuts or additions undermines trust and coordination. Clear communication keeps scope a shared commitment.

### Acknowledge Personal Risk And Professional Advice

Scope commitments may carry contractual obligations to customers or partners, with legal implications that require professional advice. This skill does not replace that counsel.

## Common Traps

### Wishlist Mistaken For Scope

A desired feature list without commitment produces chronic overruns.

### Scope Exceeding Capacity

A scope that requires more than the venture has is a setup for failure.

### No Explicit Out-Of-Scope

Without boundaries, scope creeps through accretion of small additions.

### No Forcing Function

Without pressure to commit, scope expands indefinitely.

### Ignoring Hidden Scope

Support, testing, and maintenance overhead overrun releases that counted only features.

### Clinging To Obsolete Scope

Executing a scope that no longer fits changed conditions fails by overrun.

## Self-Check

- [ ] Scope is defined as a commitment with capacity and time, not a wishlist.
- [ ] Scope matches realistic capacity and runway, including overhead.
- [ ] Explicit out-of-scope boundaries are defined for each release.
- [ ] A forcing function is used to require commitment and tradeoffs.
- [ ] Scope creep from all sources is evaluated against the current goal and boundaries.
- [ ] Core scope is built before incremental scope.
- [ ] Hidden scope, support, testing, maintenance, is accounted for.
- [ ] Scope is re-scoped when conditions change.
- [ ] Scope and changes are communicated clearly to all affected.
- [ ] Contractual obligations from scope commitments are flagged for professional legal advice.
