# MVP To V1 Roadmap

The MVP tests the riskiest assumption; the journey from MVP to v1 turns a validated assumption into a product that reliably delivers value to a defined segment. This stretch is where many ventures go wrong. Some scale prematurely, pouring growth investment into a product whose value is not yet reliable. Others stall, continuing to iterate without committing to a coherent v1 that can serve real customers. A sound MVP-to-v1 roadmap sequences the work that hardens the core value, removes the friction that blocks retention, and prepares the product to serve a growing base, without jumping to scale before the foundation is solid.

Use this skill before answering broad questions such as "what do I build after my MVP", "how do I plan my roadmap to v1", "what comes after the MVP", or "how should an entrepreneur move from MVP to a real product". The goal is to prevent the agent from treating v1 as a bigger MVP and from scaling work before the core value is reliable.

## Core Rules

### Confirm The MVP's Core Assumption Before Building V1

Before committing to a v1 roadmap, confirm that the MVP's evidence supports the core assumption: that the target segment values the solution enough to use and pay for it. If the evidence is ambiguous, the right move is more MVP iteration, not v1 investment. V1 builds on validated assumptions, not on hopes.

### Define V1 By The Segment And Value It Must Reliably Serve

V1 is not "more features"; it is a product that reliably delivers the validated value to the defined segment. Define v1 by who it serves, what value it must reliably provide, and what retention and reliability it must achieve. This definition constrains the roadmap to what serves the segment, rather than to whatever seems useful.

### Harden The Core Value Loop First

The first priority after the MVP is to harden the core value loop: make the value reliable, the action easy, and the outcome consistent. A core loop that works sometimes, for some users, cannot support growth. Invest in making the validated value dependable before extending the product.

### Remove The Friction That Blocks Retention

Identify the friction that prevents users who would value the product from reaching and staying in the core loop: confusing onboarding, slow performance, missing reliability, gaps in the workflow. Removing this friction moves retention more than adding new capabilities. Prioritize friction removal over feature addition.

### Defer Scale Work Until The Core Is Reliable

Resist the urge to build for scale, performance at millions of users, deep enterprise features, elaborate admin tools, before the core value is reliable. Scale work is expensive and premature if the product's shape is still changing. Build for the scale the next phase requires, and defer scale architecture until evidence justifies it.

### Sequence Around The Biggest Leaks

Use funnel and retention analysis to find the biggest leaks where value is lost, and sequence the roadmap to address them in order of impact. Fixing the largest leak moves the key metric more than fixing several small ones. Let the data, not opinion, set the sequence.

### Distinguish Must-Have From Nice-To-Have For V1

For each candidate v1 feature, classify it as must-have, needed for the core value to be reliable for the segment, or nice-to-have, valuable but not essential. V1 ships the must-haves; nice-to-haves defer to later versions. This discipline prevents v1 from ballooning into an endless build.

### Plan For Reliability And Support, Not Only Features

V1 must be reliable enough to serve real customers, which means investing in stability, error handling, monitoring, and support processes, not only in features. A feature-rich product that breaks or cannot be supported loses the customers it attracts. Reliability and support are v1 requirements.

### Set A Time Box To Force Commitment

V1 can expand indefinitely as more "essential" features are identified. Set a time box for v1 and ship what is ready, deferring the rest. A time box forces decisions about what truly matters for serving the segment and prevents the roadmap from becoming an open-ended build.

### Acknowledge Personal Risk And Professional Advice

V1 decisions carry stakes for venture direction, runway, and personal risk. Where the product involves data, payments, or regulatory exposure, professional advice may be needed. This skill does not replace that counsel.

## Common Traps

### Premature Scaling

Investing in growth and scale before the core value is reliable wastes runway.

### Stalled Iteration

Continuing to tweak the MVP without committing to a coherent v1.

### V1 As A Bigger MVP

Adding features without hardening the core value loop produces breadth without reliability.

### Scale Work Before Shape Is Set

Architecture for millions when the product is still changing produces rework.

### Ignoring Reliability And Support

A feature-rich product that breaks or cannot be supported loses customers.

### Open-Ended V1 Build

V1 without a time box expands indefinitely and delays serving real customers.

## Self-Check

- [ ] The MVP's core assumption is confirmed before v1 investment begins.
- [ ] V1 is defined by the segment and value it must reliably serve.
- [ ] The core value loop is hardened before the product is extended.
- [ ] Friction that blocks retention is identified and removed.
- [ ] Scale work is deferred until the core value is reliable.
- [ ] The roadmap sequence is set by the biggest leaks, not by opinion.
- [ ] Features are classified as must-have or nice-to-have, with only must-haves in v1.
- [ ] Reliability, monitoring, and support are treated as v1 requirements.
- [ ] A time box forces v1 commitment and prevents open-ended building.
- [ ] Data, payment, and regulatory exposures are flagged for professional advice.
