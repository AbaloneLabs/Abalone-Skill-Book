# Quality Control And Handoff

Editorial work moves through stages, and at each transition there is a risk: errors introduced in one stage pass unnoticed into the next, required steps are skipped under pressure, inconsistencies creep in across multiple contributors, and the final output carries the accumulated defects of the whole process. Quality control at handoffs is the discipline of verifying that work is actually ready to move, that it meets the standards of its stage, that nothing required has been omitted, and that the receiving stage receives clean, complete, consistent material. This is gatekeeping in the constructive sense: the QC editor is the last defense against errors reaching readers and the mechanism that keeps the pipeline honest. QC that is permissive or rushed lets defects through; QC that is rigorous but slow bottlenecks the pipeline. The craft is to check efficiently, to know what each stage should have produced, and to catch what matters without re-doing the work of prior stages.

## Core Rules

### Verify Stage Completion Against Defined Criteria

Each stage has entry and exit criteria. The QC editor checks the work against the exit criteria of the stage it is leaving: has development resolved the structural issues flagged; has copy-editing addressed grammar, consistency, and style; has proofreading caught the typos and formatting errors. Verification is against a defined standard, not a general impression of quality. The editor uses a checklist tied to the stage's criteria and confirms each item. Work that has not met exit criteria returns to the stage rather than moving forward, because downstream stages cannot fix upstream failures efficiently.

### Check For Errors Introduced During Editing

Editing introduces errors. A copy edit may fix a grammar issue but break a cross-reference; a style change may create inconsistency elsewhere; a deletion may leave a dangling reference. The QC editor specifically looks for introduced errors: broken cross-references, inconsistent terminology after a change, formatting disrupted by edits, meaning altered by a word change. Introduced errors are common and insidious because the editor who made them was focused on the fix, not its ripple effects. A QC pass that looks only for original errors misses the new ones.

### Confirm Consistency Across Contributors And Sections

Multi-contributor works accumulate inconsistency: different terminology for the same concept, different formatting for the same element, different treatment of similar cases. The QC editor reads across the work for consistency, checking that house style is applied uniformly, that terminology is consistent, that cross-references resolve, and that the work reads as a coherent whole rather than a patchwork. Consistency checking is especially important at handoffs where multiple contributors' work converges. The editor maintains or references the project's style decisions and enforces them across all sections.

### Verify Required Elements Are Present

Each stage and each work type has required elements: front matter, back matter, captions, alt text, cross-references, citations, disclosures, metadata. The QC editor confirms that required elements are present, complete, and correct. Omitting a required element is a common and serious failure, because the absence is invisible until a reader, regulator, or downstream system notices. A checklist of required elements for the work type prevents omission. The editor verifies presence, correctness, and completeness, not just existence.

### Reconcile Versions And Track Changes

At handoffs, version control is critical. The QC editor confirms that the correct version is being handed off, that all changes have been accepted or resolved, that track-changes and comments have been addressed, and that no contributor's edits have been lost. Version confusion, where an older draft overwrites a newer one, or where unaccepted changes persist, causes rework and errors. The editor verifies the version's integrity before accepting the handoff, ensuring the work handed forward is the work that was intended.

### Check Formatting And Production Readiness

For work moving toward publication, QC includes production readiness: formatting is correct and consistent, styles are applied rather than manual overrides, images and tables are present and correctly placed, links resolve, and the file will behave as expected in downstream systems. Production errors, broken links, missing images, misformatted tables, are caught at QC or they reach readers. The editor checks the work as it will appear to readers, not just as text, because formatting errors are invisible in a raw text view.

### Perform A Targeted Final Read For The Stage

Beyond checklist verification, the QC editor does a targeted read focused on what this stage's readers will experience. For a final pre-publication QC, this is a read for anything a reader would notice as wrong: typos, factual errors, broken references, inconsistent formatting, missing elements. The read is targeted, not a full re-edit, because the prior stages have done their work; QC is verification, not repetition. The editor reads with fresh eyes for what slipped through, focusing on high-risk areas like names, numbers, cross-references, and the first and last paragraphs readers notice most.

### Document And Communicate QC Outcomes

QC outcomes should be documented: what was checked, what was found, what was fixed, what remains. Documentation lets the receiving stage know the work's status, creates a record for accountability, and feeds back into process improvement by identifying recurring error types. The editor communicates outcomes to the relevant contributors, especially when issues are found, so contributors can improve and so the work returns cleanly for any needed fixes. Silent QC, where work passes or fails without communication, leaves contributors uncertain and processes from improving.

## Common Traps

### Permissive QC Under Schedule Pressure

Under deadline pressure, QC becomes permissive, waving work through to keep the pipeline moving. Permissive QC lets defects through that cost more to fix later or that reach readers. The editor maintains rigor on the critical checks even under pressure, accepting risk only on non-critical items and only deliberately.

### Checking Only For Original Errors

QC that looks only for errors in the original work misses errors introduced during editing. The editor specifically checks for introduced errors, broken references, inconsistencies from changes, and altered meanings, which are the signature defects of editorial work.

### Skipping The Cross-Work Consistency Check

Multi-contributor works need cross-work consistency checking, which is often skipped as time-consuming. The editor reads across sections for consistent terminology, style, and treatment, because inconsistency is a common and visible failure.

### Missing Required Elements

Omitting required elements, captions, alt text, cross-references, disclosures, is invisible until noticed and serious when it is. The editor uses a required-elements checklist for the work type and confirms each item's presence and correctness.

### Version Confusion At Handoff

Handing off the wrong version, or a version with unresolved changes or comments, causes rework and errors. The editor verifies version integrity, accepting or resolving all changes, before completing the handoff.

### Ignoring Production Readiness

For work near publication, QC must include production readiness, not just text correctness. The editor checks formatting, images, links, and downstream behavior, because production errors reach readers even when the text is correct.

### Repetition Of Prior Stages' Work

QC that re-edits rather than verifies wastes effort and bottlenecks the pipeline. The editor does targeted verification and a focused read for what slipped through, trusting prior stages' work rather than redoing it.

### Silent QC Without Communication

QC outcomes not communicated leave contributors uncertain and prevent process improvement. The editor documents and communicates what was checked, found, and fixed, feeding back into the team's development.

## Self-Check

- Has the work been verified against the exit criteria of its stage, with a checklist confirming each item?
- Has the work been specifically checked for errors introduced during editing, including broken references and inconsistencies from changes?
- Has consistency across contributors and sections been verified for terminology, style, formatting, and treatment?
- Are all required elements for the work type present, complete, and correct, confirmed against a checklist?
- Has version integrity been confirmed, with the correct version handed off, changes resolved, and no edits lost?
- For work near publication, has production readiness been checked, including formatting, images, links, and downstream behavior?
- Has a targeted final read been done for what readers would notice, focusing on high-risk areas like names, numbers, and references?
- Have QC outcomes been documented and communicated to relevant contributors, with recurring issues noted for process improvement?
- Has rigor been maintained on critical checks despite schedule pressure, with risk accepted only deliberately on non-critical items?; has the editor resisted re-editing prior stages' work, focusing QC on verification rather than repetition?
- Has the work been read as it will appear to readers, not just as raw text, to catch formatting and production errors?; would the receiving stage or final reader find this work clean, complete, consistent, and free of stage-appropriate errors?
