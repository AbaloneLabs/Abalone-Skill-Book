# Test Prioritization And Roadmap

A testing program fails not from running bad tests but from running the wrong tests. Teams gravitate toward easy changes, button-color experiments, and changes that feel safe, while the largest opportunities, structural funnel changes, pricing, and offer design, go untested. Without disciplined prioritization, a program produces a long list of small wins and little real learning. A good roadmap focuses scarce testing capacity on the changes most likely to move the business and teach something durable.

Use this skill before building a testing backlog, scoring opportunities, sequencing experiments, or reviewing a testing program's direction. The goal is to prevent the agent from spending testing capacity on low-value changes while high-value opportunities sit untested.

## Core Rules

### Prioritize By Potential Impact, Not Ease

The largest opportunities are often harder to test. Do not let ease crowd out impact.

Score by:

- the potential uplift in a business-valued metric;
- the volume of users or revenue affected;
- the strategic importance of the area;
- the confidence that the change will work;
- the learning value of the result.

Easy button-color tests win often but teach nothing and move little. Structural changes to the offer, pricing, or funnel move more and teach more.

### Ground Prioritization In Evidence Of The Problem

Do not test a change because someone had an idea. Test because evidence shows a problem.

Require:

- data showing where the funnel leaks;
- qualitative evidence of confusion or objection;
- a hypothesis explaining why the change will help;
- a sense of the size of the opportunity;
- a reason this test ranks above alternatives.

An idea without evidence of a problem is a guess. A test grounded in evidence of friction has a far higher chance of moving the metric.

### Balance Quick Wins And Strategic Bets

A roadmap of only quick wins stalls. A roadmap of only big bets starves the program of momentum.

Balance:

- quick wins that build confidence and momentum;
- strategic bets on structural changes with high upside;
- learning tests that clarify how users behave;
- risk-reducing tests on changes before full rollout;
- a cadence that sustains both learning and results.

A healthy program runs a mix. All-quick-wins programs plateau; all-big-bet programs produce too few results to sustain support.

### Sequence Tests To Respect Funnel Dependencies

Tests interact. Running them in the wrong order wastes effort or confounds results.

Sequence by:

- testing upstream changes before downstream ones they affect;
- avoiding overlapping tests on the same audience or page;
- accounting for carryover effects between tests;
- running foundational tests before optimizing details;
- isolating tests that could contaminate each other.

Changing a landing page while testing its thank-you page confounds both. Sequence so each test's result is interpretable.

### Value Learning, Not Just Winning Tests

A program that only values winning tests encourages safe, small changes. Value learning too.

Value:

- tests that disprove a strongly held belief;
- tests that reveal how users actually behave;
- tests that rule out a direction, saving future effort;
- tests that generate reusable insight;
- tests that inform strategy, not just tactics.

A losing test that kills a bad idea is valuable. A program that punishes losing tests drives the team toward trivially safe ones.

### Re-Test Surprising And High-Stakes Results

Important or surprising results deserve replication before they drive big decisions.

Replicate when:

- the result contradicts prior belief or data;
- the effect is large enough to drive major change;
- the test was run during an unusual period;
- the segment effect differs from the average;
- the decision is hard to reverse.

A single surprising test can be noise. Replication separates durable insight from coincidence.

### Maintain A Learning Roadmap, Not Just A Backlog

A backlog of test ideas is not a strategy. Maintain a roadmap of questions to answer.

Organize around:

- the big questions the program is answering;
- the sequence in which they will be addressed;
- how each test contributes to the learning agenda;
- what is known, unknown, and assumed;
- the decisions the learning will unlock.

A learning roadmap ensures each test builds on the last. A flat backlog tests disconnected ideas and accumulates little understanding.

## Common Traps

### Optimizing For Ease Over Impact

Filling the roadmap with easy changes while high-value opportunities go untested.

### Idea-Driven Testing

Testing changes because someone proposed them, without evidence of a problem.

### All Quick Wins Or All Big Bets

An unbalanced program either plateaus or starves for results.

### Overlapping And Confounded Tests

Running tests that interact so no result is cleanly interpretable.

### Punishing Losing Tests

Discouraging learning by treating only winners as valuable.

### Single-Test Decisions On Big Bets

Acting on a surprising result without replication before a hard-to-reverse decision.

### Backlog Without Strategy

A long list of disconnected test ideas that accumulates no understanding.

## Self-Check

- [ ] Opportunities are scored by potential impact, volume affected, strategic importance, confidence, and learning value.
- [ ] Each test is grounded in evidence of a real problem, not just an idea.
- [ ] The roadmap balances quick wins, strategic bets, and learning tests.
- [ ] Tests are sequenced to respect funnel dependencies and avoid overlap and confounding.
- [ ] Losing tests that disprove beliefs or reveal behavior are valued as learning.
- [ ] Surprising or high-stakes results are replicated before driving major decisions.
- [ ] A learning roadmap organizes tests around big questions, not just a flat backlog.
- [ ] The program tests structural opportunities, not only easy surface changes.
- [ ] Each test's contribution to the learning agenda is clear before it runs.
- [ ] Testing capacity is focused on the changes most likely to move the business.
