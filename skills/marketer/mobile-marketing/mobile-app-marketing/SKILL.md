# Mobile App Marketing

A mobile app is not a smaller website; it is a distinct product with its own discovery, installation, permission, and engagement economics. Mobile app marketing fails when an app is built without a clear reason to exist beyond "we should have an app", when acquisition is measured in installs rather than retained users, when permissions are requested before trust or value is established, or when push notifications are treated as a free broadcast channel rather than a fragile privilege. The cost of acquiring an install is wasted entirely if the user never opens the app a second time, which is why mobile strategy must be built around the first-session experience and the habits that follow.

Use this skill before answering questions such as "should we build an app or a mobile web experience", "how do we get more app installs", "how do we keep users coming back", "how should we use push notifications", or "how do we improve our store presence". The goal is to prevent the agent from treating an app as a checkbox deliverable and ignoring the retention and permission dynamics that determine whether it earns its place on the device.

## Core Rules

### Decide App Versus Mobile Web On Real Justification

Building an app is a major investment with an ongoing retention burden. An app must earn its existence by doing something the mobile web cannot do as well.

Justify an app by asking:

- does the product require native capabilities such as offline use, sensors, background tasks, or push?
- does value depend on frequent, habitual, low-friction access that an app provides?
- will users accept the cost of installing and keeping the app on their device?
- is the app a destination users will open repeatedly, or a one-time utility better served by web?

A one-time form, a brochure, or an occasional transaction is usually better served by mobile web. Forcing an app for low-frequency needs produces install-and-abandon outcomes and wasted acquisition spend.

### Treat Installs As The Start, Not The Goal

An install is the beginning of the relationship, not a success metric. The real metric is whether the user returns and forms a habit.

Shift focus by:

- measuring retention at D1, D7, and D30, not just install volume;
- tracking the first-session completion rate and the path to the core value moment;
- evaluating acquisition sources by downstream retention, not install cost;
- recognizing that cheap installs from low-intent sources rarely retain.

A campaign that drives thousands of installs with single-session users is a cost, not growth. Acquisition quality is only visible in retention cohorts.

### Optimize The Store Presence As A Conversion Funnel

The app store listing is a landing page, and it is often the first and only impression. A weak listing wastes every install-driving effort.

Optimize the store presence by:

- researching and selecting keywords that match real search intent;
- writing titles and subtitles that balance keyword coverage with clarity;
- using screenshots and video to communicate the core value before install;
- maintaining ratings and reviews as a trust and ranking signal;
- testing listing elements rather than assuming the first version is optimal.

Store algorithms weigh install velocity, retention, and ratings. A listing optimized only for keywords, without retention and rating support, will not sustain visibility.

### Sequence Permissions Around Value, Not Convenience

Permissions, especially notifications, location, and tracking, are granted based on trust and perceived value. Requesting them too early guarantees denial that is hard to reverse.

Sequence permissions by:

- requesting a permission only at the moment it enables a clear benefit;
- explaining why the permission is needed before the system prompt;
- never asking for all permissions during onboarding;
- treating notification permission as a high-value, carefully timed ask.

Once a user denies a permission, recovering it is difficult. The first ask must be tied to a moment where granting it makes the user's experience tangibly better.

### Build Push As A Privilege, Not A Broadcast

Push notifications are the most powerful and most easily abused mobile channel. Every irrelevant notification increases the chance the user disables notifications or uninstalls.

Use push responsibly by:

- tying each notification to genuine value or relevance for that user;
- segmenting and personalizing rather than blasting the whole base;
- respecting frequency and time-of-day preferences;
- measuring the effect of push on retention and uninstalls, not just open rate;
- making it easy for users to tune notification preferences before they disable them entirely.

A high open rate achieved through alarming or manipulative copy produces short-term engagement and long-term uninstalls. The agent should evaluate push by its effect on retained users, not campaign-level opens.

### Use In-App Messaging To Guide The Journey

In-app messages reach users at moments of attention inside the product. They are powerful for onboarding, feature discovery, and contextual offers.

Use in-app messaging by:

- triggering messages from user behavior and context, not arbitrary schedules;
- keeping messages relevant to the current screen and task;
- avoiding interruption of core workflows;
- using onboarding tooltips to drive the first value moment;
- reserving promotional in-app messages for moments of earned attention.

### Connect Acquisition, Retention, And LTV

Mobile app economics only work when acquisition cost is recovered through retained user value. The full system must be connected.

Connect by:

- measuring LTV by acquisition source and cohort;
- ensuring CAC payback occurs within a realistic retention window;
- feeding retention insights back into acquisition targeting;
- refusing to celebrate install growth that produces poor retention.

## Common Traps

### Building An App Without Justification

Creating an app for low-frequency or web-suitable needs produces install-and-abandon outcomes and wasted investment.

### Celebrating Installs Over Retained Users

Install volume from low-intent sources hides poor retention and destroys unit economics.

### Requesting Permissions Too Early

Asking for notifications, location, or tracking before establishing value guarantees denials that are hard to recover.

### Treating Push As A Free Broadcast

Blasting irrelevant notifications drives notification disabling and uninstalls, destroying the channel for everyone.

### Store Listing Optimized For Keywords Only

Ignoring ratings, retention, and creative quality undermines the ranking signals the algorithm rewards.

### First Session That Never Reaches Value

Users who do not experience core value in the first session rarely return, making onboarding the real retention battleground.

### Acquisition Evaluated On Install Cost Alone

Cheap installs that do not retain are a cost dressed up as growth.

## Self-Check

- [ ] The decision to build an app is justified by native capabilities or habitual use the web cannot match.
- [ ] Success is measured by D1, D7, and D30 retention, not install volume.
- [ ] Acquisition sources are evaluated by downstream retention and LTV, not install cost.
- [ ] The store listing is treated as a conversion funnel with tested keywords, creative, and ratings.
- [ ] Permissions are requested at moments of clear value, never all at once during onboarding.
- [ ] Push notifications are personalized, frequency-controlled, and measured by retention impact and uninstall risk.
- [ ] In-app messages are behavior-triggered and relevant to the current context, not interruptive.
- [ ] The first session is engineered to reach the core value moment quickly.
- [ ] CAC payback is evaluated within a realistic retention window by cohort.
- [ ] Install growth that produces poor retention is treated as a cost, not success.
